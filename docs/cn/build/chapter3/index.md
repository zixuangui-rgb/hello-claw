# 第三章 提示词系统——Agent的"灵魂工程"

> **核心问题**：如果一个 Agent 能被任意指令改变性格、随意遗忘约定、每次对话都从零开始，它还算"你的" Agent 吗？OpenClaw 的提示词系统，正是为了回答这个问题而设计的。

---

## 从临时指令到持久人格

### 传统提示词的困境

你可能有过这种体验：花了一个小时调教好 AI 的回复风格，第二天打开新对话，它又变回了出厂设置。

根源很简单：**传统提示词是临时的**。它只存在于当次对话的上下文中，对话结束即消散——重复劳动、风格漂移、无法积累，这三个麻烦不是 Bug，而是"指令与对话活在同一生命周期"的必然结果。

更深的问题在于：当这个 Agent 工作得很好时，你很难说清楚"好"在哪里。是哪一段提示词措辞恰到好处？是哪个约束条件发挥了关键作用？没有记录，没有结构，就没有可复现的工程知识——每次调教都是从零开始的即兴创作。

这不是效率问题，这是**可工程化**的问题。一个不可复现的过程，无法被团队共享，无法被系统性改进，也无法在出现问题时进行根因分析。

### 范式突破：配置即灵魂

OpenClaw 的解法触及问题本质：**把提示词写进文件，永久保留**。

```
传统方式：
  对话开始 ──→ 用户输入提示词 ──→ AI 响应 ──→ 提示词随对话消亡

OpenClaw 方式：
  ┌─────────────────────────────────────┐
  │  SOUL.md / USER.md / AGENTS.md ...  │  ← 持久存在，与对话无关
  └──────────────────┬──────────────────┘
                     ↓ 每次推理前自动读取，动态组装
  每次对话 ──────────→ 系统提示词 ──→ AI 响应
```

这个区别的意义远不止"方便"。它实现了一个根本性解耦：**Agent 的身份与单次对话脱钩**。Agent 的人格、记忆、行为规范写在文件里，独立于任何一次对话而存在——可以版本控制、可以团队共享、可以持续迭代。

传统"系统提示词工程"把所有配置压进一段文本，注入对话开头。这种做法有三个隐患：**密度过高**（人格定义、用户偏好、工作流程全混在一起，改动一处可能影响意想不到的另一处）；**职责模糊**（Agent 行为异常时，无法判断是哪个层面出了问题）；**无法协作**（一段私有文本，谁改了什么、为何改，无从追溯）。

OpenClaw 把这段"大杂烩"拆分成八个职责清晰的文件。每个文件只负责一件事——这是软件工程"单一职责原则"在 Agent 配置层面的具体实践。

这也意味着 Agent 的"身份"终于有了可追溯的历史。你可以看到三个月前的 SOUL.md 和今天的差异，理解为什么当初做了某个调整，在新项目里复用一套经过验证的配置——这些在"提示词即临时文字"的时代都是不可能的。持久化的本质，是让 Agent 的成长成为一种**可积累的资产**，而不是每次对话结束后归零的临时状态。

这个范式转变的深层含义，要等到你真正开始管理多个 Agent、或者与团队共同维护一个 Agent 时才会充分显现。当某个 Agent 的行为需要审计，当你想把一个表现优秀的 Agent 配置迁移到新项目，当两个人需要同时修改不同方面的配置而不互相干扰——这些场景下，"配置在文件里"和"配置在对话里"之间的差距，是量级上的区别。

## 八份档案——各司其职

从一段混合文本到八个结构化文件，这个拆分不只是"更整齐"，它为 Agent 配置引入了**可寻址性**：当行为异常时，你知道去哪里找根源；当需要调整时，你知道改哪个文件会影响什么。下面逐一解析每个文件的职责边界，以及那些容易混淆的关键区分。

### 总览

```
┌──────────────┬────────────────────────────┬──────────┬──────────────┐
│ 文件         │ 核心作用                   │ 类比     │ 加载时机     │
├──────────────┼────────────────────────────┼──────────┼──────────────┤
│ SOUL.md      │ 人格、价值观、行为边界     │ 性格     │ 每次对话     │
│ USER.md      │ 用户画像与偏好（静态）     │ 用户手册 │ 每次对话     │
│ AGENTS.md    │ 工作流程与决策规则         │ 操作手册 │ 每次对话     │
│ TOOLS.md     │ 环境信息备忘录             │ 资源清单 │ 每次对话     │
│ IDENTITY.md  │ Agent 名称、版本等基础身份 │ 身份证   │ 每次对话     │
│ MEMORY.md    │ 长期记忆——事实与经验      │ 日记本   │ 每次对话     │
│ HEARTBEAT.md │ 定期检查清单，驱动主动性   │ 闹钟提醒 │ 每次对话     │
│ BOOTSTRAP.md │ 新工作区初始化引导         │ 入职培训 │ 仅首次，用后归档 │
└──────────────┴────────────────────────────┴──────────┴──────────────┘
```

八个文件，八个职责，互不交叉。理解它们的边界划定逻辑，比记住每个文件的用途更重要。

从另一个角度看，这八个文件覆盖了 Agent 运作所需的所有维度：**它是谁**（SOUL、IDENTITY）、**它服务谁**（USER）、**它怎么工作**（AGENTS）、**它在哪里工作**（TOOLS）、**它记得什么**（MEMORY）、**它怎么主动行动**（HEARTBEAT）、**它如何初始化**（BOOTSTRAP）。没有冗余，没有遗漏——这种完备性不是偶然的，而是对 Agent 运作机制系统性拆解的结果。

下面重点解析其中几组容易混淆的文件对：它们往往在用途上有表面相似性，但承担的职责截然不同。

### SOUL.md 与 AGENTS.md：人格与规则的分离

LLM 会根据对话语境自动调整扮演状态——回答技术问题时扮专家，回答感情问题时扮朋友。学术上称为**角色漂移（Role Drift）**，是模型的结构性特点而非缺陷。但对于需要**可预测行为**的 Agent，角色漂移是致命的。`SOUL.md` 被放在系统提示词的最高优先级位置，目的只有一个：无论语境如何变化，锁定角色。

一个写得好的 SOUL.md 包含三层定义：**我是谁**（核心身份，不可动摇的定位）；**我怎么说话**（沟通风格，让每次回应有辨识度）；**我不做什么**（行为边界，把高风险情景提前决策掉，不留给运行时的随机性）。这三层缺一不可——没有边界定义的人格配置，遇到极端情况时仍然会漂移。

```
SOUL.md 回答"这个 Agent 是什么样的存在"：
  "我是一个专注安全的系统工程师，在明确之前我会保守行事。"
  → 价值观与信念，无法被用户请求覆盖，是底层操作系统

AGENTS.md 回答"这个 Agent 怎么工作"：
  "收到部署请求时：检查测试覆盖率 → 确认环境变量 →
   发送确认消息 → 等待批准 → 执行部署。"
  → 决策流程与操作规范，可根据场景调整
```

人格决定"怎么想"，规则决定"怎么做"。两者分离，才能分别优化。

### USER.md 与 MEMORY.md：两种不同的"了解你"

这两个文件常被混淆，但描述的是完全不同维度的信息：

```
USER.md（你是什么样的人——静态画像）
─────────────────────────────────
  技术水平：中级开发者
  偏好：简洁直接，不要废话
  语言：中文优先

MEMORY.md（我们经历过什么——动态积累）
─────────────────────────────────────
  2026-03-10：确认用 pnpm，不是 npm
  2026-03-12：部署前必须先跑 db:migrate
  凤凰项目 = 客户 X 的电商重构项目
```

USER.md 是相对稳定的用户画像；MEMORY.md 是随使用不断积累的经历记录。混淆两者会导致 USER.md 随时间越来越长、越来越难维护。MEMORY.md 会随时间增长，但上下文有限，超出上限的部分会从尾部截断——**重要的记忆写在前面**，定期清理过时条目是 Agent 维护的一部分。

MEMORY.md 的维护节奏很重要：不是每次对话后都要更新，而是在以下时机主动整理——当你意识到某个约定已经改变（比如从 npm 切换到 pnpm 了），当某个项目代号可以归档了，当文件已经接近截断阈值时。把 MEMORY.md 视为一份有生命力的档案，定期修剪，保持它的精准性，是比"尽量多记"更好的管理策略。

### 其余四份档案

**TOOLS.md** 看到名字，很多人以为它控制"哪些工具可以用"。实际上它是 Agent 的**环境备忘录**——当前项目里有哪些具体资源：服务器地址、项目路径、凭证位置等。就像给一个经验丰富的工程师看的那张"环境速查表"——他知道怎么做，但不知道你们的服务器 IP 在哪。真正控制工具可用性的是工具配置文件，详见第四章和第七章。

TOOLS.md 最大的价值，是把那些"当然要知道但不可能每次都口头告知"的环境信息固化下来。数据库连接字符串在哪找、哪个目录是前端构建产物、哪个环境变量文件不能提交——这些信息如果不写进 TOOLS.md，每次对话都要重新交代，或者 Agent 猜错了你还要纠正。

**IDENTITY.md** 存放 Agent 名称、版本等基础身份信息，是最稳定的一份档案。

**HEARTBEAT.md** 定义了一个定期检查清单：在特定时间节点或触发条件下，Agent 主动检查待处理任务、提醒用户、关注异常。大多数 Agent 是被动的，HEARTBEAT.md 让它从"问答机器"进化为"主动协作者"。

**BOOTSTRAP.md** 仅在工作区创建后**首次运行时**加载，完成后归档，永不再出现。它的角色是入职引导员：建立 IDENTITY.md、收集用户偏好填充 USER.md、写入最初的 MEMORY.md 条目。一旦引导完成，它功成身退——初始化逻辑不应永远留在系统中。

## 系统提示词的组装

配置文件写好只是第一步，它们如何被读取、以什么顺序组装成系统提示词、哪些在什么情况下会被加载——这些机制决定了配置文件如何真正影响 Agent 的行为。理解组装机制，才能理解"为什么要这样写配置"，以及"配置生效或不生效的根本原因"。

### 每次推理前重新组装

当你发出一条消息，Agent 推理之前，系统会**重新读取所有配置文件**，动态组装系统提示词：

```
系统提示词（从上到下，优先级递减）：
┌─────────────────────────────────────────┐
│  工具定义（可用工具列表）               │ ← 始终最前，不可被配置覆盖
│  安全护栏                               │
│  技能列表（名称 + 描述 + 路径，非全文） │ ← 索引，按需加载
│  工作目录信息                           │
│  八份配置文件内容                       │ ← SOUL / USER / AGENTS / TOOLS / ...
│  沙箱限制（如启用）                     │
│  当前时间                               │ ← 始终最后，最低优先级
└─────────────────────────────────────────┘
```

越靠前的内容越难被后面的内容覆盖——这个顺序反映了一种优先级哲学，不是随意安排的。每次都重新读取文件，意味着修改文件后**下一条消息即刻生效**，不需要重启任何进程。

这个组装顺序也暗示了一个实用的调试思路：当 Agent 行为出现预期外的结果，可以沿着优先级链从上往下排查——是工具定义层的问题，是安全护栏覆盖了你的配置，还是 SOUL.md 与 AGENTS.md 之间存在冲突？分层结构让排查有路可循，而不是面对一段混合文本无从下手。

### 截断原则与两种加载模式

每个配置文件有大小上限，所有文件合计也有总上限。超出部分从**尾部截断**，并向 Agent 注入截断警告：

```
文件内容（从头到尾）：
┌──────────────────────────────────────────────┐
│  最核心的规则和人格定义   ← 永远在上限之内   │
│  常用的工作流程           ← 通常在上限之内   │
│  详细的边界情况说明       ← 可能被截断       │
│  历史记录和参考信息       ← 经常被截断       │
└──────────────────────────────────────────────┘
                  ↑ 截断发生在这里
```

**把最重要的内容写在文件开头**——这不是建议，是需要内化成习惯的配置原则。

截断机制还有一个容易被忽视的含义：它定义了文件的"有效内容预算"。如果你发现一个配置文件已经很长，在往里添加内容之前，先问自己两个问题：这条新内容比文件末尾现有的内容更重要吗？有没有可以删除的过时条目为它腾出空间？配置文件的维护，本质上是一个持续优化的资源分配问题。

Agent 在不同场景下加载的内容也不同：

```
┌──────────────────┬──────────────────┬───────────────────────────────────┐
│ 模式             │ 触发场景         │ 加载内容                          │
├──────────────────┼──────────────────┼───────────────────────────────────┤
│ 完整模式（full） │ 正常用户对话     │ 所有八份文件 + 技能列表 + 心跳配置 │
│ 轻量模式（min）  │ 子 Agent 调用    │ 仅 AGENTS.md + TOOLS.md           │
└──────────────────┴──────────────────┴───────────────────────────────────┘
```

子 Agent 执行的是父 Agent 委派的具体任务，不需要完整人格定义，不需要了解用户是谁，不需要长期记忆——只需要知道"能做什么"（AGENTS.md）和"有哪些资源"（TOOLS.md）。上下文窗口有限，每个字符都有成本。不相关的上下文是噪音，它增加推理成本，还会干扰 Agent 对任务的专注度（LLM 在过长上下文中注意力会分散，这是有研究记录的模型特性）。

这个设计揭示了一个更深层的架构原则：**在 Agent 系统里，上下文不是越多越好，而是越精准越好**。主 Agent 在做协调和判断，需要完整的身份与背景；子 Agent 在做执行，只需要操作手册和资源清单。角色不同，所需信息不同，强行统一只会造成干扰。

轻量模式让主 Agent 下可以挂载数十个专业子 Agent，每个子 Agent 专注自己的 AGENTS.md 和 TOOLS.md，不会因为互相加载彼此的"人格配置"而产生干扰——这是清晰的**关注点分离**：协调逻辑在主 Agent，执行逻辑在子 Agent，两层之间只通过任务指令和结果交互，而非共享上下文。

### 技能的懒加载机制

技能文件（`skills/*.md`）不会全文注入系统提示词，只注入一行索引：

```
┌─────────────────────────────────────────────────────────────────┐
│ 注入到系统提示词（索引）：                                      │
│   deploy-app: 将应用部署到目标环境  [path: skills/deploy-app.md]│
│   run-tests:  执行测试并生成报告    [path: skills/run-tests.md] │
└─────────────────────────────────────────────────────────────────┘
          │ Agent 判断需要某个技能时
          ↓
┌─────────────────────────────────────────┐
│ 主动用 read 工具读取技能全文（按需）    │
│   skills/deploy-app.md  ← 此时才读取   │
└─────────────────────────────────────────┘
```

这是"索引—按需加载"模式，和数据库索引的设计思想如出一辙：基础提示词保持精简，同时支持数十个技能共存而不撑爆上下文。

技能懒加载的另一个好处是**技能库可以无限扩展**而不影响基础性能。无论你有 5 个技能还是 50 个技能，注入系统提示词的索引部分大小几乎不变。Agent 只在推理到需要某个技能时才读取其全文——这是对上下文资源的精准管理，而不是粗放式的全量加载。

组装机制的整体设计哲学是一致的：**按需加载，重要的在前**。工具定义在最前是因为安全边界不能被覆盖，技能全文按需是因为大多数技能在多数对话中用不到，截断从尾部开始是因为文件末尾的内容通常优先级最低。每一个设计决策都有对应的权衡逻辑，理解这些逻辑才能在自定义配置时做出正确的判断。

## 热更新与配置即代码

配置写进文件，打破了 Agent 身份与对话的绑定；热更新机制，进一步打破了"修改配置"与"重启系统"之间的绑定。这两个解耦合在一起，让 Agent 开发真正进入了工程化的快速迭代模式。

### 热更新意味着什么

传统软件修改配置需要：提交代码 → CI/CD → 构建 → 部署 → 验证。快则几分钟，慢则半小时。

调教一个 Agent 是反复实验、快速迭代的过程。如果每次修改都要等一个部署周期，实验的节奏就彻底被打断了。OpenClaw 的热更新机制保证：**修改配置文件，保存，发下一条消息，新配置即刻生效**。没有重启，没有部署，没有等待。

热更新解锁了一种全新的工作方式：在对话过程中即时调整 Agent 的行为。发现 Agent 的解释风格太过简洁？暂停对话，在 SOUL.md 里加一句"遇到复杂概念时主动举例说明"，保存，继续对话——下一条回复就会体现这个变化。这种即时反馈让配置调整从"工程任务"变成了"实时对话的延伸"。

### 文件指纹比对：轻量的感知机制

热更新最朴素的实现是"每次都重新读取所有文件"——但这意味着无谓的磁盘 IO（大多数时候文件根本没变）。OpenClaw 用的是**文件指纹比对**：

```
首次加载：
  读取文件内容 → 记录指纹（路径 + 大小 + 修改时间）→ 存入缓存

后续每次对话前：
  快速比对新旧指纹
    ├── 指纹相同 ──→ 直接用缓存（零额外开销）
    └── 指纹变了 ──→ 重新读文件，更新缓存，新配置立即生效
```

指纹只记录路径、大小和修改时间，不读取文件内容本身，比对速度极快。文件未变时几乎零开销，文件变化时立即感知——一个优雅的平衡。这种设计体现的是一种普遍的工程原则：缓存失效的时机要精准，既不能过于保守（每次都重读），也不能过于激进（永不更新）。文件指纹正好落在这个甜点区间。

### 为什么这在工程上很重要

把 SOUL.md 里的"严谨的工程师"改成"幽默的编程老师"，保存文件，发下一条消息——Agent 的沟通风格立刻切换，不需要任何其他操作。这把 Agent 调教从"改→部署→测试→改"的瀑布循环，变成"改→看效果→改"的实时反馈循环。

更深层的意义是：**配置文件是代码**。既然 Agent 的身份存在于文件中，版本控制、历史追溯、分支实验、团队协作——软件工程的全套工具都可以直接套用。你可以用 Git 对 Agent 的演化过程进行版本化管理：知道它从哪里来，到哪里去，每一次改变是为了什么。快速反馈循环加上版本控制，是调教高质量 Agent 的必要条件。

热更新与版本控制是互补的：热更新解决"快速实验"的问题，版本控制解决"可回溯、可协作"的问题。前者让迭代速度够快，后者让迭代方向有记录。两者结合，Agent 配置的演化过程才是真正可管理的工程活动，而不只是个人经验的随意积累。

## 好配置的特征

配置文件写得好不好，直接决定 Agent 行为是否可预测、可信任。一个常见误区是把配置文件当作"写给人看的文档"——把意图、原则、愿景堆在一起，期待模型自行理解。但 LLM 不是在"理解你的意图"，而是在"匹配可执行的行为规则"。好配置的核心特征，是让模型没有歧义地知道在任何给定情况下该做什么。

以下三个特征，是区分"好配置"与"感觉上写了很多但没什么用的配置"的核心维度。

### 具体优于抽象

LLM 处理模糊指令时会"自由发挥"，而自由发挥的结果往往不符合预期：

```
❌ 模糊：
  "请你谨慎行事"
  → "谨慎"的解释权交给了模型，行为不可预测

✅ 具体：
  "执行任何删除或修改文件的操作前，先列出受影响的文件路径，
   等我用'确认'或'ok'回复后再执行"
  → 定义了可观测的行为，结果可预测
```

配置文件里的每一条"抽象原则"，都应该有对应的"具体行为描述"做支撑。

一个实用的检验标准：把你写的配置条款交给另一个人，问他"如果 Agent 遇到 X 情况，按照这条规则，它会做什么？"如果对方的回答和你预期的一致，说明这条规则够具体；如果对方说"不确定"或者给出了不同的解读，说明这条规则还停留在抽象层面，需要进一步细化。写配置不是写诗，歧义是配置文件的敌人。

### 边界定义：负向规范同样重要

大多数人写配置时关注"Agent 应该做什么"，但"Agent 不应该做什么"同样重要，有时更重要。

边界定义的本质是**提前决策**：把那些"需要 Agent 在高压场景下做判断的高风险情景"，在配置层面提前处理掉，不把决策权留给运行时的随机性。一个可信任的 Agent，往往有明确的边界清单——它知道自己的能力范围，知道哪里是禁区，知道何时应该停下来询问。

边界写得越清晰，Agent 在模糊地带的表现越可预测。"不主动删除数据库记录，除非用户在同一条消息中明确写出表名和条件"——这样的边界条款，比任何关于"谨慎"的抽象原则都更有效。它不依赖模型的道德直觉，而是用可验证的行为规则替代了主观判断。

### 文件布局与持续迭代

**布局原则**（与截断机制配合）：

```
文件开头  ──→  最核心的规则与人格定义（必须在截断线之上）
前三分之一──→  常用工作流程
中部      ──→  详细边界情况说明
末尾      ──→  历史记录、参考信息（可接受被截断）
```

布局不只是技术要求，也是对"哪些内容真的核心"的强迫性思考。当你不得不把最重要的内容放在前面时，你必须先想清楚：什么是最重要的？这个约束本身就在推动你把配置写得更有层次感。

Agent 配置不是一次性完成的。你无法在第一天就预想到第三十天会遇到的所有问题。配置的完善需要真实的使用经验——每当 Agent 行为出乎意料，把这次经验提炼成规则写进配置；每当 MEMORY.md 积累了过时条目，定期清理并把最重要的提升到前面。这是从**隐性知识到显性知识**的提炼过程，也是 Agent 持续成长的核心路径。

实践上，建议把 Agent 配置目录作为项目仓库的一部分纳入 Git 管理。每次对配置的重大修改都附上提交说明，解释为什么做这个改变——哪次行为异常触发了这条规则的添加，哪次工作流优化导致了 AGENTS.md 的重构。这些说明不是给 Git 看的，是给未来的你或者接手这个 Agent 的团队成员看的。一个有完整提交历史的 Agent 配置仓库，本身就是一份关于这个 Agent 成长历程的活文档。

三个特征——具体优于抽象、边界同样重要、内容按重要性排布——背后是同一个底层逻辑：把"需要在运行时做的判断"尽可能前置到配置层面。配置文件是 Agent 运作之前能做的最后一道工程工作，这道工作做得越扎实，运行时的不确定性就越少。

## 本章小结

将 Agent 身份从对话生命周期中解耦，并将其存储在版本控制文件中，OpenClaw 把"AI 配置"从一次性的、随对话消散的临时艺术，变成了可积累、可迭代、可协作的持久工程。这不是技术细节的优化，而是思维方式的根本转变：同一套让软件系统可靠的工程实践——单一职责、版本控制、快速反馈循环——现在可以直接应用于 Agent 行为本身。配置文件不是 Agent 的"说明书"，而是 Agent 本身的一部分；修改配置，就是修改 Agent 的认知与行为边界。

这个转变的架构意义在于：Agent 的可预测性不再依赖于用户每次对话时的"精心提示"，而是固化在文件结构里。人格、记忆、规则各归其位，职责边界清晰，行为异常时可以精确定位是哪一层出了问题。配置文件纳入 Git，Agent 的演化有了历史、有了可回溯的轨迹——它不再是一个黑盒，而是一个可以被工程化管理的系统。

这也改变了 Agent 开发的时间感。Agent 不再是一个在某天"配置好"然后使用的静态工具，而是一个随时间持续成长的协作者。每一次行为出乎意料，是一次把隐性知识提炼成显性规则的机会；每一次成功的工作流，是一次值得提炼进 AGENTS.md 的经验积累。管理 Agent 配置，就像管理一个团队成员的成长记录——你既是用户，也是培养者。

- **配置即灵魂**：持久化文件取代临时提示词，Agent 身份独立于单次对话而存在
- **八份档案，职责分明**：SOUL 锁定人格、AGENTS 规定流程、USER 描述画像、MEMORY 积累经验、HEARTBEAT 驱动主动性——每个文件的修改影响范围可预测
- **SOUL 与 AGENTS 的分离**：人格决定"怎么想"，规则决定"怎么做"，两者分离才能分别优化
- **USER 与 MEMORY 的分离**：静态画像与动态积累不可混淆，否则两个文件都会因此退化
- **优先级布局**：组装顺序与文件内部的内容布局，都遵循"重要的在前"原则，与截断机制配合
- **两种模式服务两类场景**：完整模式服务正常对话，轻量模式服务子 Agent——最小必要上下文是可扩展性的基础
- **技能按需加载**：索引注入，全文按需读取，提示词精简而能力无限扩展
- **热更新的实验价值**：文件指纹缓存让配置修改立即生效，实时反馈循环加速 Agent 调教
- **Agent 是演化的活文档**：配置随真实使用持续迭代精炼，从隐性知识到显性知识的提炼是 Agent 成长的核心路径
