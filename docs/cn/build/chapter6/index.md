# 第六章 统一网关——一个入口，连接所有渠道

> **核心问题**：OpenClaw 怎样把 Telegram、Discord、Slack、WhatsApp 等不同平台的消息统一接入，并在多渠道下依然保持同一个 Agent 的连续协作体验？

---

上一章讲的是消息循环与事件驱动，那一章解决的是：**消息进来以后，系统怎样排队、怎样执行、怎样不串线。** 但还有一个同样重要的问题没有展开：**这些消息到底从哪里来？**

如果 Agent 只活在一个聊天窗口里，这个问题很简单。可 OpenClaw 想做的不是“一个网页里的 AI 助手”，而是一个真的能接进现实渠道的 Agent。你可能在 Telegram 找它，同事可能在 Discord 里找它，另一个团队可能在 Slack 里接它。平台一多，系统要面对的就不只是“用户说了什么”，还包括：

- 这条消息来自哪个平台；
- 这个平台怎么认证；
- 这个平台的用户 ID 长什么样；
- 这条回复该发回哪里；
- 同一个人换个平台，Agent 还能不能认出来；
- 不同平台能力不一样，回复该怎么展示。

如果这些事情都直接塞进 Agent 核心里，核心很快就会变得又重又乱。所以这一章真正要讲的，不是“怎么接某个平台的 Bot”，而是：**OpenClaw 为什么要专门放一个 Gateway 在最外层，以及它怎样把多渠道世界整理成一个统一入口。**

这一章的主线可以先记成一句话：**统一网关的价值，不只是多接几个平台，而是把平台差异挡在外面，让 Agent 内部始终只处理统一的“人、会话、消息”。**

---

## 第一节 为什么 OpenClaw 需要统一网关

这一节只回答一个问题：**为什么 OpenClaw 不能让每个平台直接连到 Agent，而必须先经过一层统一网关。**

### 1.1 这不只是“多接几个平台”，而是避免系统被平台细节拖垮

很多同学第一次听到“统一网关”时，会下意识觉得它像一个高级附加功能：“我现在只用 Telegram，是不是根本用不上？”“这不就是以后多接平台时再考虑的事吗？”这个理解只对了一半。统一网关当然和多平台有关，但它真正要解决的是：**渠道差异会不会一路污染到系统核心。**

先看最表层的差异：

| 平台 | 用户 ID 长相 | 常见文本字段 | 认证方式 |
|------|--------------|--------------|----------|
| Telegram | 一串数字 | `message.text` | Bot Token |
| Discord | Snowflake ID | `message.content` | Bot Token / OAuth |
| Slack | `U...` 这样的成员 ID | 事件对象里的嵌套字段 | App Token / OAuth |
| 飞书 | `open_id` 或 `user_id` | 更深的 JSON 结构 | tenant token |

这还只是“长得不一样”。更麻烦的是，私聊和群聊结构不同、回复旧消息的方法不同、线程和按钮支持度不同、单条消息长度限制也不同。没有统一网关，每接一个新平台，就得往 Agent 核心里再塞一批平台专用逻辑。

短期看，这些 `if-else` 都能跑；长期看，系统会越来越像一团打结的线。所以统一网关的第一层意义不是“让架构显得高级”，而是：**把平台差异拦在最外层，不让它们一路流进 Agent 核心。**

### 1.2 平台孤岛最麻烦的，不是字段不同，而是身份被拆碎了

如果只是字段不一样，其实还只是工程麻烦。真正会直接影响协作体验的，是**身份割裂**。想象一个常见场景：你今天在 Telegram 跟 Agent 聊了很久，告诉它你的项目、偏好、语言习惯和最近关注的任务；第二天你换到 Discord 找它，如果系统没有做身份关联，Agent 看到的就不是“还是你”，而是“一个全新的平台 ID”。

于是就会出现一种很别扭的感觉：**你明明没变，Agent 却像第一次见你。** 这就是平台孤岛真正伤人的地方。它不是把消息分散到了不同平台，而是把“同一个人”拆成了很多份。

这里最好先分清三个很容易混在一起的词：

| 词 | 先理解成什么 |
|----|--------------|
| 渠道 | 你从哪个平台来，比如 Telegram、Discord |
| 会话 | 当前这段具体对话现场 |
| 身份 | “这个人到底是谁” |

这三层不是同一回事。如果系统分不清它们，就会出现两种典型错误：**该合的时候没合**，同一个人换个平台认不出来；**不该合的时候乱合**，不同场景的上下文被硬拼到一起，结果对话串线。

所以统一网关要解决的，从来不只是“消息怎么收”，还包括：**同一个人跨平台出现时，系统能不能认出他；不同场景的对话，又能不能继续保持隔离。**

### 1.3 统一网关真正保护的，是 Agent 核心的纯度

走到这里，就能更准确地理解 Gateway 的角色了。它不是一个“消息转发器”那么简单，更像一点的说法是：**Gateway 是 Agent 的感官系统，也是外部世界和内部推理之间的翻译层。**

可以把它想成下面这样：

```text
平台原始事件
→ Gateway 先读懂它
→ 统一成系统内部认识的消息
→ 再交给 Agent 去判断和处理
```

反过来也一样：

```text
Agent 先产出统一形式的回复
→ Gateway 再看当前平台能做到什么
→ 把它翻译成这个平台最合适的输出
```

这意味着 Agent 核心可以长期只关心几件事：谁在找我，现在是哪段会话，用户说了什么，我应该怎么回复，任务下一步怎么推进。至于“这是 Telegram 还是 Discord”“用户 ID 是数字还是字符串”“平台支不支持按钮”“回复该不该流式输出”“超长消息该怎么切”，这些都留在 Gateway 这一层处理。

这件事的意义在于，它给系统划出了一条清楚的边界：**网关负责理解外部世界，Agent 负责理解任务本身。** 边界一清楚，后面的演化就轻松很多：升级 Agent 推理能力，不必碰每个平台的接入逻辑；新增渠道，不必改 Agent 核心；优化某个平台的展示效果，也不必牵动全部系统。

如果把第一节收成一句话，就是：**统一网关存在的意义，是把“平台差异、身份映射、输出适配”这些外部复杂性，统一挡在 Agent 核心之外。**

## 第二节 统一网关是怎样跑起来的

这一节要回答的问题是：**一条来自外部平台的消息，进入 OpenClaw 以后，到底会经历哪些步骤，最后又怎样变成一条合适的回复发回去。**

### 2.1 第一步：先把平台“方言”翻成系统能听懂的话

不同平台的原始事件，很像不同地区的方言。它们都在表达“有人发来一条消息”，但表达方式完全不同。所以 Gateway 做的第一件事，不是路由，也不是回复，而是：**标准化。**

这里先把“标准化”讲成人话：就是把长相完全不同的原始输入，整理成系统内部统一认识的一种结构。不管消息来自哪里，系统最后都希望拿到类似这样的信息：渠道是谁，用的是哪个账号，发消息的人是谁，这条消息来自哪个聊天对象，文本内容是什么，有没有附件、引用、线程信息。

可以把这一步压成下面这条链路：

```text
平台原始事件
→ 平台插件读取并解析
→ 提取出统一字段
→ 形成系统内部的标准消息
```

这里第一次出现一个很重要的词：`ChannelPlugin`。你可以先把它理解成：**某一个平台对应的一块适配器。** 它的工作很像翻译：入站时，把平台自己的事件格式翻成系统统一格式；出站时，再把系统统一回复翻回平台认识的格式。

这就是为什么 OpenClaw 官方会把平台能力组织成插件，因为平台差异本来就应该被关在插件里，而不是散落在 Agent 各处。用更直白的话说：**Agent 不应该学会十种平台方言。Agent 只需要学会一种“系统普通话”，剩下的交给插件。**

### 2.2 第二步：消息标准化以后，还要先决定“该交给谁”

标准化完成后，系统才真正开始做路由。这里先把“路由”讲人话：**路由就是决定，这条消息到底该交给哪一个 Agent 来处理。**

这一步为什么必须单独存在？因为在 OpenClaw 里，系统未必只有一个 Agent。你可能会有一个负责个人事务的 Agent，一个负责开发协作的 Agent，一个只在某个群里工作的 Agent，甚至不同账号、不同频道，默认接不同 Agent。所以网关收到消息以后，不能直接扔给“唯一的那个 Agent”，因为很多时候根本不存在“唯一”。

从机制上看，这一层对应的就是路由解析逻辑。你不需要死记函数名，但最好知道它大致在做什么：**系统会根据当前渠道、账号、对话对象，以及你写下的 `bindings` 规则，从更具体的匹配一路走到更通用的匹配，最后找到最合适的 Agent。**

可以先把常见概念分清：

| 名字 | 可以先理解成什么 |
|------|------------------|
| `channel` | 哪个平台，比如 Telegram、Discord |
| `account` | 这个平台上的哪个机器人账号 |
| `peer` | 这条消息来自哪个聊天对象，比如某个私聊、群、频道 |
| `bindings` | 你手动写下的“这类消息交给哪个 Agent”规则 |

可以把路由过程想成这样：

```text
先看有没有非常具体的绑定
→ 再看这个渠道/账号有没有默认归属
→ 再看更通用的回退规则
→ 最后决定由哪个 Agent 接手
```

这一层的关键不是“规则多不多”，而是：**消息必须先交对人，后面的一切才有意义。** 如果路由错了，就会出现个人消息跑去找团队 Agent、某个群里的问题被发给错误工作流、本该隔离的渠道意外复用了别的默认逻辑。

所以第六章讲网关时，一定不能只讲“平台怎么接进来”，还要讲“接进来以后到底交给谁”。这是统一网关真正的中枢工作之一。

### 2.3 第三步：找到 Agent 还不够，还要把消息放进正确的会话里

很多人第一次理解到这里时，会觉得：“既然已经知道交给哪个 Agent 了，不就可以开始处理了吗？”还不够。因为“交给哪个 Agent”和“落到哪段会话里”，是两件事。

这里最重要的词是 `sessionKey`。你可以先把它理解成：**系统用来标记‘这条消息到底属于哪段会话’的钥匙。** 为什么需要这把钥匙？因为同一个 Agent 下面，可能同时存在很多段对话现场：你和它的私聊、同一个群里的另一段线程、别的平台上你的另一段私聊、其他用户和它的并行对话。

如果系统不先算出正确的 `sessionKey`，这些消息就可能掉进同一个上下文里，结果就是上一章讲过的“串线”。所以从机制上讲，Gateway 在这一层做的是：

```text
已经知道由哪个 Agent 接手
→ 再根据渠道、账号、对话对象、身份映射等信息
→ 算出这条消息属于哪段 session
→ 然后才送去后面的消息循环
```

这也是为什么 Gateway 和上一章的消息循环是连着的。上一章解决的是“同一条 lane 里的消息怎么排队”；这一章在它前面补上了另一半：**什么叫“同一条 lane”，是谁来决定的。**

这里还必须讲清楚两个特别容易混的东西：`sessionKey` 和 `identityLinks`。

| 名字 | 它主要解决什么 |
|------|----------------|
| `sessionKey` | 这条消息应该落到哪段会话里 |
| `identityLinks` | 不同平台上的 ID 是否属于同一个人 |

它们有关，但不是一回事。更准确地说，`identityLinks` 解决“你是谁”，`sessionKey` 解决“你这次在哪段现场里说话”。举个很短的例子就明白了：假设你在 Telegram 和 Discord 上都绑定成了同一个身份，这意味着系统可以知道**这两个平台上的两个 ID，本质上是同一个人**；但这并不等于系统就一定要把两边聊天记录合成一段会话。

很多时候，正确做法恰恰是：身份可以共用，会话依然分开。因为你在 Telegram 和 Discord 上说的，可能根本不是同一件事。另外还有一个和私聊场景相关的小开关：`dmScope`。你可以先把它理解成：**它决定私聊消息在多细的粒度上拆会话。** 这类配置看起来不起眼，但会直接影响你最后的协作体验：粒度太粗，容易串线；粒度太细，本来该连续的对话又会被拆碎。

所以第六章真正想让你明白的是：**统一网关不是只负责“接消息”，它还负责把消息放进正确的身份和正确的会话里。**

### 2.4 第四步：Agent 给出统一回复后，网关还要把它翻回各个平台

前面讲的是消息怎么进来，现在来看另一头：回复怎么出去。OpenClaw 在内部通常会先形成比较统一的回复表达。从用户视角，你可以先把它理解成：**Agent 先想清楚自己要说什么，再由 Gateway 决定在不同平台上怎样说出来。**

这一层最重要的设计思想叫：**优雅降级。** 第一次听到这个词，可以这样理解：如果平台能力更强，就尽量用更好的展示；如果平台能力没那么强，也至少保证内容完整可达。

比如同样是一条回复，不同平台能做的事情可能完全不同：

| 能力 | 有的平台能做什么 | 没有的平台怎么办 |
|------|------------------|------------------|
| Markdown / 富文本 | 渲染标题、列表、代码块 | 降成清楚的纯文本 |
| Streaming | 一边生成一边更新消息 | 改成一次性发完整结果 |
| Typing Indicator | 显示“正在输入” | 没有就静默等待 |
| Actions / Buttons | 给出可点击按钮 | 改成文字说明 |
| Threads | 在原线程继续回复 | 退化为普通消息回复 |

这就是 Gateway 为什么不能只是“原样转发文本”。因为统一网关要处理的，不只是内容本身，还包括：当前平台支持什么能力，当前账号打开了哪些能力，这条回复该不该流式输出，长消息需不需要自动切分，当前回复要不要沿用最近一次有效回路。

这里再解释一个容易忽略的小点：`lastRoute`。你可以先把它理解成：**系统会在会话里记住‘上一条有效回复是从哪条回路出去的’。** 这样做的好处是，当同一段会话继续往下走时，系统更容易把后续回复送回正确的渠道和账号，而不是突然跑偏。

另外，长消息切分也常常在这一层处理。因为不同平台对单条消息长度限制不一样，如果系统不先切好，用户就可能看到发送失败、半条消息丢失、代码块被从中间硬切开。所以一个成熟的 Gateway，不只是“能发出去”，而是要做到：**发得出去、发得对、发得像这个平台本来就该有的样子。**

### 2.5 第五步：优雅降级不只是“显示不同”，还包括长消息和失败处理

很多同学一看到“优雅降级”，脑子里先想到的都是样式问题，比如：

- 有的平台显示卡片；
- 有的平台只能显示纯文本；
- 有的平台支持按钮；
- 有的平台只能打一段普通消息。

这当然是优雅降级的一部分，但还不止这些。

在真正运行时，Gateway 还要处理另外两类很现实的问题：

1. **这条回复太长了，平台一次发不下；**
2. **这次发送失败了，但失败的原因未必一样。**

先看第一类，长消息切分。

不同平台对单条消息长度的限制并不一样。  
如果系统不在出口侧先处理，你很容易遇到三种糟糕体验：

- 消息直接发送失败；
- 只发出前半段，后半段丢失；
- 代码块或列表被从中间硬切开，读起来很难受。

所以 Gateway 在这一层通常不只是“发一条消息”，而是会先判断：

- 当前平台大概能接受多长的文本；
- 这条回复需不需要切成多段；
- 切分时怎样尽量保住段落边界、代码块边界和语义完整性。

再看第二类，失败处理。

发送失败并不都意味着同一种问题。有些是暂时的，比如网络抖动、平台限流；有些是结构性的，比如账号没配好、渠道没启用、内容不被平台接受。

这时 Gateway 真正要做的，不是简单地“报错完事”，而是先判断：

- 这是临时问题，还是永久问题；
- 值不值得重试；
- 重试会不会带来重复发送或更大混乱。

你不用记所有实现细节，但最好记住这个原则：

**优雅降级不只是“富文本降成纯文本”，还包括“平台不理想时，系统仍然尽量把内容安全、完整、可预期地送达”。**

### 2.6 第六步：为什么说 `ChannelPlugin` 让系统真正具备了可扩展性

走到这里，统一网关的主链路已经基本清楚了：

```text
平台事件
→ 插件解析
→ 标准化
→ 路由
→ 计算 sessionKey
→ 进入消息循环
→ Agent 产出回复
→ 插件按平台能力渲染并发回
```

最后还差一个关键问题：**为什么 OpenClaw 能不断接新平台，而不是每接一个平台就得重写系统？** 答案就在 `ChannelPlugin` 这层插件边界上。

从设计角度看，插件机制最大的价值不是“代码组织更好看”，而是：**系统终于可以把‘平台特有能力’和‘Agent 通用能力’拆开演化。**

对一个新平台来说，最小要做的事情通常并不多：

1. 告诉系统自己是谁；
2. 能把平台事件读进来；
3. 能把系统回复发回去。

在这个基础上，才会继续叠加更丰富的能力，比如流式输出、typing 指示、按钮或交互动作、线程回复、特定平台附件处理。这意味着接入新平台的成本被控制在一个合理范围里。你不需要先把所有高级能力一口气做完，先把“能收、能发、能跑通”做好，系统就已经能工作；后面再按平台特性慢慢增强。

这就是很典型的工程化思路：入口门槛低，上限又足够高。也是因为有了这层插件注册和能力声明，Gateway 才能真正做到：**新增平台时，主要改插件，不改 Agent 核心。**

如果把第二节收成一句话，就是：**统一网关的运行机制，本质上是一条“标准化 → 路由 → 会话归属 → 平台化输出”的流水线，而 `ChannelPlugin` 则让这条流水线可以持续扩展。**

## 第三节 知道机制以后我们应该如何做

前两节讲的是“为什么存在”和“它怎么运转”，现在更实际的问题来了：**如果你真的要开始用 OpenClaw 的 Gateway，该怎么配，才比较稳，比较不容易掉坑。**

### 3.1 第一步，不是一下接很多平台，而是先跑通一条最小链路

很多初学者最容易犯的错误，是一上来就想把所有渠道都接好：Telegram 一起配，Discord 也顺手接上，Slack 反正也要用干脆一起做，还想顺手把 streaming、thread、按钮全打开。这种做法最容易把排错难度直接拉满。

更稳的做法其实非常朴素：**先选一个你最熟、最常用、资料最清楚的平台，把最小链路跑通。** 所谓“最小链路”，至少包括四件事：

1. 能收到消息；
2. 能正确路由到目标 Agent；
3. 能进入正确会话；
4. 能把回复再发回正确的地方。

只要这四步跑顺了，这一章的大部分核心机制其实就已经活起来了。这时候你再去接第二个平台，排错会轻松很多，因为你已经知道：Agent 核心本身是通的，问题大概率出在新增渠道的适配或配置上。

所以零基础同学最好的起手式不是“全开”，而是：**单渠道先通，多渠道后加。**

### 3.2 配置时优先想清楚四件事：谁来接、谁是谁、哪段算同一会话、能力开到哪一步

真正开始配 Gateway 时，建议你先把下面四个问题写下来。

#### 第一问：这类消息应该交给谁处理

这对应的是路由层。你要先明确：哪些渠道默认归哪个 Agent，哪些频道或群组要单独绑定，哪些账号只服务某一类任务。如果这一层没先想清楚，后面再怎么调显示效果都没意义。

#### 第二问：哪些平台 ID 其实是同一个人

这对应的是 `identityLinks`。这里最关键的不是“绑得越多越好”，而是：**只在你真的确定是同一人的时候才绑定。** 因为一旦绑错，系统就会把本不该共享的偏好或身份信息混到一起。

#### 第三问：哪些消息应该继续算同一段会话

这对应的是 `sessionKey` 和 `dmScope` 相关思路。这里一定要记住一句话：**同一个人，不等于同一个会话。** 所以你要先判断：是希望不同渠道各自保留上下文，还是希望某些场景合并得更紧一些，私聊要拆得多细，群聊要不要按线程隔离。

#### 第四问：平台增强能力要不要一开始全打开

通常不建议。更稳的思路是分两层：

| 阶段 | 优先目标 |
|------|----------|
| 第一层 | 先保证能收、能发、不串线 |
| 第二层 | 再去打开 streaming、threads、actions 等增强能力 |

因为“能用”和“好用”是两件事。先把“能用”做稳，后面才有资格讨论体验优化。

### 3.3 一定要学会区分“身份共享”和“上下文共享”

这是第六章里最容易被误用的一点，所以值得单独讲。很多人一看到跨平台身份统一，就会自然觉得：“既然 Telegram 和 Discord 上都是我，那聊天历史是不是也应该自动打通？”不一定。

更常见、也更稳的做法通常是：

- **身份可以共享；**
- **上下文往往还是按会话隔离。**

原因很简单：“认出你是谁”和“把两段不同场景的聊天揉成一段”是两件不同风险的事。前者通常能提升体验；后者如果做错，很容易让现场变脏。

可以用这张表快速判断：

| 你想共享的是什么 | 更适合怎么处理 |
|------------------|----------------|
| 用户偏好、称呼、长期习惯 | 可以依赖身份关联共享 |
| 某个平台里这几轮具体聊了什么 | 更适合留在当前会话里 |
| 某个群里的上下文 | 通常按群或线程隔离 |
| 不同平台上的私聊内容 | 默认分开通常更稳 |

所以如果你未来要做跨平台协作，最重要的不是“有没有 `identityLinks`”，而是：**你心里要先清楚，自己到底想共享哪一层。**

### 3.4 多渠道体验优化，最好按“先保底，再增强”的顺序来

Gateway 很容易让人上头，因为一旦看到 streaming、按钮、线程回复、富文本卡片、typing 指示，你会很想全部打开。但从工程上讲，更推荐的顺序其实很朴素：

```text
先保证内容一定送达
→ 再保证会话一定正确
→ 再保证展示效果更好
→ 最后才追求平台特有增强能力
```

这背后的原因很简单：用户最先感知到的，不是“有没有流式输出”，而是：

- 回没回我；
- 回对地方没；
- 会不会串对话；
- 同一个人能不能认出来。

这些基础问题不稳时，越花哨的增强能力，越容易让你更难排障。

所以优雅降级不仅是系统内部设计原则，也应该成为你的配置原则：

**先把所有平台都做成“至少稳稳能用”，再去追求“在某个平台上特别好用”。**

### 3.5 遇到问题时，按固定顺序排错，效率会高很多

Gateway 相关问题看起来五花八门，但很多时候可以按固定顺序收敛。建议你优先这样查：

先看下面这个固定顺序：

1. 先看消息有没有真的进来。  
先确认渠道插件是否启动，平台事件有没有成功送到本地，对应账号是否配置正确。如果消息压根没进系统，后面就不用看了。

2. 再看路由是不是命中了正确 Agent。  
如果消息进来了，但去了错误的 Agent，通常要优先检查：`bindings` 写没写对，默认账号和默认归属是不是配错，某条更具体的规则有没有把它截走。

3. 再看会话是不是算对了。  
如果 Agent 对人是对的，但上下文明显串线或断裂，就要看：`sessionKey` 的粒度是不是不合适，`dmScope` 是否把私聊拆得太粗或太细，`identityLinks` 是否把不该合的人绑在一起。

4. 最后再看平台能力和出站适配。  
比如，为什么没有 streaming，为什么按钮没显示，为什么长消息被切得不好看，为什么回复回到了错误渠道或错误位置。这时再去看平台能力声明、出站适配和最近一次回路信息，思路会清楚很多。

可以把这条顺序记成一句很短的话：

```text
先看进没进来
→ 再看交没交对人
→ 再看会话算没算对
→ 最后再看显示和体验
```

对零基础同学来说，这种固定顺序特别重要，因为它能帮你避免一上来就陷进细节里乱翻配置。

## 本章小结

这一章表面上讲的是 Gateway，实际上讲的是：**OpenClaw 怎样把外部世界整理成 Agent 能稳定处理的统一入口。**

最重要的理解可以收成下面几条：

1. **统一网关不是“多平台附件”，而是系统边界。**  
它把平台差异、身份映射、输出适配这些复杂性挡在外层，避免污染 Agent 核心。

2. **网关处理的不只是消息格式，还包括“交给谁”和“落到哪段会话”。**  
路由决定由哪个 Agent 接手，`sessionKey` 决定进入哪段上下文，这两步缺一不可。

3. **身份统一和会话统一不是一回事。**  
`identityLinks` 解决“是不是同一个人”，会话隔离解决“是不是同一段现场”，这两层最好分开理解。

4. **优雅降级是统一网关的关键设计。**  
平台能力强，就展示得更完整；平台能力弱，也要保证内容完整送达，而不是直接失效。

5. **`ChannelPlugin` 让系统真正能扩展。**  
新平台主要通过插件接入，Agent 核心不用跟着一起改，这才是长期可维护的多渠道架构。

到这里，OpenClaw 的“感官系统”就清楚了。它负责把外面五花八门的平台世界，翻译成内部统一的协作现场。

但只会接消息、发消息还不够，因为 Agent 一旦真的能动手，就会立刻碰到另一个更硬的问题：**能力越强，边界怎么守住？**

这就是下一章要讲的内容。
