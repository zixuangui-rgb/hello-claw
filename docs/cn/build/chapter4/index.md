# 第四章 工具系统——Agent的"手脚"

> **核心问题**：四个基础工具为何足够？LLM如何理解并选择工具？如何通过组合创造无限能力？

---

## 一、最小完备集——四个工具为何够用

### 工具设计的权衡

给 Agent 配置工具，是一道没有完美答案的权衡题。工具太多，LLM 面临选择压力，边界模糊，准确率下降；工具太少，任务无法完成；工具太专，维护成本急剧攀升。

这个权衡在实践中有一个常见的陷阱：每当遇到工具能力不足的情况，最直觉的反应是"再加一个工具"。但工具集的扩张是有成本的——不只是实现成本，更是认知成本。每增加一个新工具，就需要确保其描述与已有工具的边界不产生歧义；每增加一个与已有工具功能相似的工具，就需要在描述中明确两者的区分依据，否则这个新工具很可能让 LLM 在已有工具和新工具之间产生混淆，导致两者都用得不准确。工具集的健康度，不由工具数量衡量，而由工具间的正交性和描述清晰度衡量。

OpenClaw 的解法是：**四个足够通用的基础工具，加上技能（Skills）按需扩展**。这不是折中，而是经过深思熟虑的最小完备设计。其精神来源是 1978 年 Bell Labs 的 Unix 哲学——"让每个程序只做一件事，并把它做好；让程序能够协作。"少即是多，在工具设计领域，这不是口号，而是可验证的工程结论。

工具数量与准确率之间存在一个非线性关系：在工具集较小时，增加工具会提升能力；超过某个临界点后，工具边界的模糊开始主导，LLM 的选择失误率上升，整体效果反而下降。四个通用工具加技能的架构，恰好落在这条曲线的高效区间——足够强大，足够清晰。工具集的健康度，不由工具数量衡量，而由工具间的正交性和描述清晰度衡量。

这里还有一个常被忽视的认知负担问题。LLM 在每次工具选择时，需要在所有可用工具的描述中做检索和匹配。工具集越大，这个检索空间越大，边界相似的工具之间的混淆概率越高。将高频任务组合沉淀为专用工具确实能减少调用步骤，但同时也在工具集里引入了更多相似功能的工具，代价是边界模糊和选择混乱。技能机制恰好提供了一个出口：用知识扩展代替工具扩展，在不增加工具集复杂度的前提下提升任务完成质量。

认知负担的问题在工具边界相似时尤为突出。考虑一个场景：工具集中同时有 `read`、`read_code`、`read_log` 三个工具，前者是通用的，后两者是针对特定文件类型优化的。每次 Agent 需要读取文件时，都要先判断"用哪个 read"——这个判断本身消耗了推理资源，而且在文件类型不明确时会产生选择犹豫。与其引入三个 read 工具，不如让 `read` 内部自动检测文件类型并采用最优策略——这样工具集保持简洁，智能留在工具内部而不是要求 LLM 在调用时做额外决策。

### 感知-推理-行动的闭环

四个工具并非随意凑数，它们对应智能体与世界交互的三种基本动作：

注意这个闭环图有一个值得深思的细节：图中只有一个感知工具（read），但有三个行动工具（write、edit、exec）。这不是不对称——而是真实反映了智能体与世界交互的本质：感知是通用的（读取信息）；行动是多样的（创建、修改、执行，三种截然不同的操作语义）。将这三种操作语义合并为一个工具，不是在减少工具数量，而是在制造模糊——你没有减少复杂度，只是把它藏到了工具内部。

```
┌──────────────────────────────────────────────┐
│                                              │
│   感知世界 ──→  read（读取信息，理解现状）      │
│                      ↓                       │
│   思考推理 ←──  LLM（理解情况，制定计划）      │
│                      ↓                       │
│   改变世界 ──→  write（创建新文件）            │
│                edit（修改现有文件）            │
│                exec（执行命令，触达外部）       │
│                      ↓                       │
│   观察结果 ──────────────────→ 回到感知       │
│                                              │
└──────────────────────────────────────────────┘
```

| 工具 | 核心能力 | 在闭环中的角色 |
|------|---------|--------------|
| `read` | 读取文件内容，获取信息 | 感知——了解世界现状 |
| `write` | 创建新文件 | 行动——在世界中创造 |
| `edit` | 精确修改已有文件 | 行动——在世界中改变 |
| `exec` | 运行命令，连接外部系统 | 行动——触发外部效应 |

四个工具的角色划分清晰呈现了一个设计意图：只有 `read` 承担感知职责，`write`、`edit`、`exec` 都是行动工具。这意味着 Agent 的所有感知输入都经过同一个受控通道——`read`，这使得对感知输入的统一处理（如截断、清理、格式化）成为可能，也让审计"Agent 看到了什么"变得简单。将感知与行动的通道分离，是工具设计中另一处值得关注的架构决策。

这个闭环每转一圈，Agent 就完成一个"感知-思考-行动"周期。缺少感知，Agent 只会盲目行动；不把工具返回作为新的感知输入，Agent 就无法根据实际情况调整计划。值得注意的是，`write`、`edit`、`exec` 虽然是行动工具，但每次调用的返回同样是新的观察——成功或失败的信息都会流回 LLM 的推理过程，驱动下一步决策。这是工具系统与普通函数调用的根本区别：函数调用是过程性的，工具调用是对话性的——每一次调用都是 Agent 与世界之间的一轮对话，而不是单向的命令发出。

### 为什么不能"只用 exec"

`exec` 能运行 `cat`、`echo`、`sed`，理论上能替代其他三个工具。但专用工具提供了两件 `exec` 给不了的东西：

```
┌─────────────────┬──────────────────────────┬────────────────────────────┐
│ 操作            │ 只用 exec                 │ 专用工具                   │
├─────────────────┼──────────────────────────┼────────────────────────────┤
│ 读文件          │ 意图模糊，无法区分"读取"  │ read：意图明确，安全策略    │
│                 │ 与"执行"                  │ 可以据此差异化             │
├─────────────────┼──────────────────────────┼────────────────────────────┤
│ 写文件          │ 崩溃时文件可能损坏        │ write：原子性写入，不存在   │
│                 │                          │ 中间破损状态               │
├─────────────────┼──────────────────────────┼────────────────────────────┤
│ 修改文件        │ 可能误改所有同名字符串    │ edit：精确匹配，找不到就    │
│                 │                          │ 报错，不会静默误改         │
└─────────────────┴──────────────────────────┴────────────────────────────┘
```

专用工具是对常见操作的**安全封装**，不是对 `exec` 的重复，而是对它的补充。更深层的理由是意图的可见性：当 LLM 调用 `read`，系统知道这是一次只读操作，可以对其应用更宽松的安全策略；而调用 `exec cat file.txt` 在意图层面与 `exec rm -rf /` 没有任何区别。专用工具把意图编码进了工具名称本身，让系统能在不解析命令内容的情况下做出差异化的安全决策。

"只用 exec" 的方案在技术上可行，但它把本该由工具系统承担的结构性保证，下沉到了命令字符串的层面。命令字符串是 LLM 生成的概率性产物，而工具系统的结构保证——原子写入、精确匹配——是确定性的。用概率机制去替代确定性机制，是一个方向性的错误，不管成功率有多高。

## 二、四大工具深度解析

### read：智能读取，保护上下文

LLM 的上下文窗口是有限的。把一个 10MB 的日志文件直接读进来，上下文就满了，Agent 会忘记自己在做什么。`read` 工具的核心价值在于它的**三层防御机制**：

```
┌────────────────────────────────────────────────────┐
│                  read 三层防御                      │
├────────────────────────────────────────────────────┤
│  第一层：预算感知                                   │
│  ├─ 计算当前上下文剩余空间                          │
│  └─ 文件超出预算 → 触发截断机制                     │
├────────────────────────────────────────────────────┤
│  第二层：智能截断（按文件类型差异化）               │
│  ├─ 代码文件 → 保留头部（import + 函数签名）        │
│  ├─ 日志文件 → 保留尾部（最新错误最有价值）         │
│  └─ 配置文件 → 原则上完整保留（配置项相互依赖）     │
├────────────────────────────────────────────────────┤
│  第三层：分页支持                                   │
│  ├─ 明确告知还有多少内容未读                        │
│  └─ Agent 可发起第二次调用继续读取（懒加载）        │
└────────────────────────────────────────────────────┘
```

三层防御的本质，是把对上下文的保护从外部强制约束变成工具内部的自律机制。`read` 还会自动清理噪音：Base64 图片数据替换为占位说明，超长行截断提示，二进制内容发出警告——让上下文空间用在真正重要的地方。

这里有一个重要的设计权衡：截断必然损失信息，但不截断会让 Agent 因上下文耗尽而丢失更重要的信息。`read` 的差异化截断策略——代码保留头部、日志保留尾部——是基于文件类型的信息价值分布做出的工程判断，而非简单地截断到固定字节数。Agent 对时效性也需要保持意识：在关键决策前重新感知（执行破坏性操作前重读文件，提交大范围修改前重跑测试）是健康的工作节奏，用少量额外感知成本换取更高的行动准确性。

分页机制是三层防御中最容易被低估的一个。它的核心贡献不是"能读大文件"，而是"主动告知还有多少没读"。这个透明度让 Agent 能够做出知情决策：当前读到的内容是否足够？是否需要发起第二次调用？这比静默截断（截断了但不告知）要好得多——静默截断会让 Agent 以为已经看到了全部内容，从而基于不完整信息做出过于自信的决策。信息的完整性与信息的新鲜度一样，是 Agent 决策质量的基础条件。

### write：原子性创建

`write` 创建新文件，最重要的特性是**原子性写入**：先写入临时文件，写入成功后再重命名为目标文件。重命名在操作系统层面是原子的——要么完成，要么不完成，不存在中间损坏状态。在 Agent 任务可能被随时中断的场景下，这是保证文件完整性的基本手段。

另外两个辅助特性：**路径自动创建**（父目录不存在时自动创建）、**编码智能处理**（自动处理 BOM 头、平台换行符差异），让 Agent 无需为底层细节分心。`write` 的职责边界是明确的：只用于创建全新文件。这条约束看似限制，实则是保护——它防止了因文件名拼写错误导致意外覆盖已有内容的场景，同时让工具调用历史中的每一次 `write` 都保持清晰的语义——全新创造，而非替换。

`write` 与 `edit` 之间的边界也是刻意设计的。`write` 负责创建，`edit` 负责修改，两者的职责不重叠。这种分离让 Agent 在描述自己意图时必须明确"我是在创建一个新的东西"还是"我是在修改一个已有的东西"。这不只是工具层面的区分，也是任务意图层面的区分——一个意图清晰的工具调用，在审计时比一个意图模糊的调用要有价值得多。

### edit：精确修改，失败安全

`edit` 的设计哲学可以用一句话概括：**宁可报错，不可误改**。

`edit` 要求提供要替换的旧内容和新内容，在文件中搜索精确匹配，找到才替换，找不到就返回错误。这与 `sed` 截然不同——`sed` 会替换所有匹配的字符串，可能误改多处；而 `edit` 在找到唯一匹配时才执行替换，匹配到多处时同样报错，要求 Agent 提供更多上下文以消除歧义。

"找不到就报错"是一种宽容的设计——它宽容的是错误，而不是误操作。Agent 收到报错后，可以重新 `read` 文件确认实际内容，再重新尝试。精确匹配还带来**可追溯性**——审计工具调用历史时，能明确看到 Agent 用什么替换了什么，而不是一个模糊的"对某个模式执行了替换"。在多步骤任务中，这种可追溯性是事后分析 Agent 决策路径的重要依据。

`edit` 设计中最值得学习的一点是：它把"失败"变成了有用的信息，而不是障碍。当 `edit` 报告"找不到指定内容"，这个报错本身传递了两个高价值信息：文件当前状态与 Agent 的记忆不一致（可能有外部修改），以及 Agent 提供的旧内容片段不够精确（可能需要提供更多上下文）。这两个信息都能直接指导下一步操作，比一个"替换失败，原因未知"的报错要有用得多。工具的错误设计，应该和工具的成功路径设计一样认真——好的错误信息不是"操作失败"，而是"为什么失败，应该怎么做"。

### exec：连接外部世界的桥梁

`exec` 是四个工具中权力最大的一个。它能运行任何 Shell 命令，触达几乎任何外部系统：测试套件、外部 API、服务管理、数据库、构建脚本……凡是命令行能做的，`exec` 都能做。工具的能力边界不由 `exec` 自身决定，而由第七章讨论的安全机制在外层定义。这种分离是有意为之：工具专注功能，安全专注边界，两者解耦，各司其职。在开发阶段可临时放宽限制，生产部署时再收紧到最小权限——工具本身不需要为此维护不同逻辑。

`exec` 是工具系统中的"开放接口"——它的能力范围由底层系统和权限决定，而不由工具本身限制。这种开放性是双刃剑：它让 Agent 能够触达任何命令行可达的系统，同时也意味着任何通过 `exec` 发起的操作都需要在安全层面被仔细审计。理解这一点对于设计 Agent 系统至关重要：不应该通过限制 `exec` 的功能来实现安全，而应该通过外层的安全机制来界定哪些命令允许被执行。功能与安全的分离，让两者都能被独立优化和调整。

两个值得关注的技术特性：
- **伪终端（PTY）支持**：模拟真实终端环境，处理颜色代码、实时输出、交互式提示，让 Agent 能与期望终端的程序正常交互。没有 PTY，许多程序会检测到非终端环境而改变行为——禁用颜色输出、缓冲所有输出直到结束、甚至拒绝启动交互式模式。
- **后台进程模式**：像开发服务器这样持续运行的命令，可在后台启动立即返回，Agent 随时可查看输出或终止它。这让 Agent 能在服务运行的同时继续执行其他任务——启动服务、修改代码、重新测试，构成一个自然的开发节奏。

`exec` 的返回处理同样重要：完整返回 stdout 和 stderr，区分退出码正常与异常，这些细节共同决定了 Agent 能否从每次命令执行中获得充分的诊断信息。`exec` 是四个工具中最容易在返回处理上偷懒的——只返回退出码，或者只返回最后几行输出。这种偷懒在短期内显得"干净"，但长期来看，它让 Agent 在遇到复杂错误时只能盲目重试，而不是基于完整信息做出有效诊断。工具的返回设计，值得与工具的功能设计同等认真对待。

## 三、工具描述即接口

### LLM 理解工具的方式

**LLM 选择工具主要依赖工具的描述文本，而不是工具的内部实现**。描述写得好，选择准确率高；描述含糊，选择就会出错。工具描述不只是给人类读的文档，它是 LLM 实际使用的接口规范。

可以用一个类比来理解：函数签名告诉编译器如何调用函数，但它不能告诉调用者"什么时候应该调用这个函数，什么时候不应该"。工具描述承担的正是这个角色——它不仅说明工具"能做什么"，更要说清楚"在什么情况下该用，在什么情况下不该用"。一个只描述功能、不说明边界的工具描述，相当于一个没有文档的公共 API——在使用者对领域不熟悉时，很难做出正确的调用决策。

写工具描述是一项工程实践，不是文档写作。好的描述应该主动消歧义：当两个工具的功能存在重叠时，描述需要明确说明两者的区分依据；当工具有重要的前置条件时，描述需要把前置条件说清楚；当工具的副作用可能影响后续步骤时，描述需要把副作用列明。描述的质量直接映射到工具选择的准确率，进而影响整个任务的成功率。在一个有多个相似工具的系统中，描述是否清晰往往比工具功能是否强大更重要。

有一个简单但实用的测试方法来评估工具描述质量：把描述给一个不熟悉系统的人读，问他"这个工具在什么时候应该用，什么时候不应该用"。如果他能给出清晰且正确的回答，描述质量是合格的；如果他感到困惑，或者给出了与设计意图不符的答案，描述需要改进。这个测试之所以有效，是因为 LLM 理解工具描述的方式，在语义层面与人类理解文字的方式高度相似——只要描述对有相关背景的人类是清晰的，对 LLM 大概率也是清晰的。

一个好的工具描述需要包含三个要素：**功能**（这个工具做什么）、**限制**（不能做什么，有什么副作用）、**适用场景**（什么情况下用这个而不是别的工具）。描述里缺少边界，会导致 LLM 在模糊地带随机选择，产生难以复现的问题。

### 工具名称的力量

工具名称是描述的第一个词，也是最重要的一个词。`read` 与人类阅读的概念对齐——安全、单向、只看不改。`write` 对应创作——产生新事物。`edit` 对应修改——在已有基础上调整。`exec` 对应执行——触发动作，产生效果。四个名字都是直觉性的，让 LLM 在没读完描述之前，就对功能有大致正确的预判。这是**意图对齐**的刻意追求，不是偶然。

工具名称与 LLM 预训练语料中对这些词的理解高度一致，让自然语言推理能够直接迁移到工具选择上。一个被命名为 `file_content_retriever` 的工具，在功能上与 `read` 相同，但它削弱了名称与自然语言概念的对齐，让 LLM 在选择时需要额外的认知开销来建立映射。工具的信任边界同样在描述层体现——`read`、`write`、`edit` 的职责划分本身就是最小权限原则的体现：`read` 是只读工具，即使实现上允许，写入操作也不应出现在其代码路径中。安全策略在外层定义（详见第七章），工具描述则是安全意图在 LLM 可见层面的第一道表达。

工具名称的选择还涉及到 LLM 对工具"安全性"的直觉判断。LLM 在选择工具时，会将名称与其对"风险"的理解挂钩——`read` 类的操作直觉上是低风险的，`exec` 类的操作直觉上是需要谨慎的。这种直觉与工具实际的安全级别对齐，有助于 LLM 在面对需要权衡时做出更保守、更合理的选择。工具命名是系统设计中最廉价的安全机制，也是最容易被忽视的一个。

工具描述随着系统演进是需要持续维护的。新增工具后，原有工具的描述可能需要更新以消除新的歧义；任务场景发生变化后，某些"适用场景"的说明可能需要调整。把工具描述视为不变的基础设施是一个危险的假设——它实际上是一个随系统演进而演进的接口合同，值得定期审查和更新，就像接口文档一样。工具描述的衰退往往不是因为描述被改错，而是因为系统的其他部分发生了变化，导致原有描述不再准确反映当前的工具使用边界。

## 四、组合性与反馈机制

### 原语与组合

`read`、`write`、`edit`、`exec` 是 OpenClaw 中的四个**原语**（primitive）——最基本、不可再分解的操作单元。单独看，每个原语功能简单；其价值在于**组合性（Composability）**。

这里可以用编程语言的基本类型来做类比。整数、字符串、布尔值本身都是极其简单的，但由它们组合出的数据结构和算法可以表达任意复杂的计算。工具系统的原语与此类似：`read`、`write`、`edit`、`exec` 各自只做一件简单的事，但由 LLM 编排的工具链可以完成任意复杂的任务。简单性不是功能的缺陷，而是组合能力的前提——你无法用功能复杂的工具构建出功能简单工具的那种组合灵活性。

一个典型的重构任务，可以拆解为以下动态工具链：

```
搜索引用 → 读取上下文 → 修改定义 → 修改引用处 → 运行测试 → 确认结果
exec      read        edit        edit         exec       read
```

这个序列没有被预编程，而是由 LLM 根据任务目标在运行时动态生成。每一步完成后，LLM 读取结果，决定下一步。这是 ReAct 模式在工具层面的直接体现：推理驱动行动，行动产生新的观察，观察触发新的推理。"代码迁移"这类复杂任务，全程只用这四个工具：读取旧代码（read）→ 分析依赖（exec）→ 创建新结构（exec）→ 逐文件改写（read + write/edit）→ 更新引用（exec + edit）→ 运行测试（exec）→ 修复失败（read + edit）→ 确认结果（read）。整个过程中，没有任何新工具被引入。

动态编排与静态工作流的核心差异在于：面对意外情况时的处理能力。静态工作流在步骤 N 失败时，要么重试，要么终止，没有能力根据失败的具体原因动态调整后续步骤。动态编排下，LLM 读取 N 步的失败详情，理解根本原因，可能决定插入一个额外的诊断步骤，或者从一个不同的角度重新尝试，或者向用户请求更多信息。这种弹性是四个原语工具与 LLM 动态推理结合后自然涌现的能力——不是预先设计进去的，而是从工具的原子性和推理的灵活性中自然生长出来的。

四个原语组合出的能力，远超各自能力的总和——这是涌现的典型例子。组合性与工具数量无关，与工具的正交性强相关。正交意味着每个工具只做自己的事，没有功能重叠，没有隐式依赖。正交性越高，工具越容易被自由组合而不产生意外的交叉影响；组合层次越高，涌现出的能力越强大。工具的原子性、正交性、组合性，不只是单 Agent 的优良属性，也是多 Agent 协作的基础条件——当多个 Agent 共享同一组基础工具时，它们的行为边界是可预期的，协作产生的副作用是可追溯的。

组合性的一个实际表现是：Agent 能够通过多步骤工具链实现工具集中没有直接对应项的操作。例如，"安全地重命名一个被多处引用的函数"不是任何单个工具的直接能力，但通过 `exec`（搜索引用）+ `read`（确认上下文）+ `edit`（逐处替换）+ `exec`（运行测试验证）的组合序列，完整实现了这个操作，且每一步都是可审计的。这种"通过组合实现不存在的工具"的能力，是基于原语工具的系统相对于基于专用工具的系统的核心优势之一——你不需要为每一类操作预先准备一个专用工具。

```
┌──────────────────────────────────────────────────────┐
│  组合能力的演化层次                                   │
├──────────────────────────────────────────────────────┤
│  原子操作   单个工具调用（read / write / edit / exec）│
│      ↓                                               │
│  工具链     多工具顺序调用，完成有内在逻辑的任务片段  │
│      ↓                                               │
│  单 Agent   工具链动态编排，完成完整任务              │
│      ↓                                               │
│  多 Agent   多个 Agent 分工协作，各自使用工具集       │
└──────────────────────────────────────────────────────┘
```

### 工具调用的双重角色——观察即决策

把 Agent 工具理解为"执行手段"只抓住了一半。另一半同等重要：**每一次工具调用的返回值，都是 ReAct 循环的新一轮"观察"**。

```
┌──────────┐    调用     ┌──────────┐    返回     ┌──────────────────┐
│          │ ──────────→ │          │ ──────────→ │ 观察（新的感知）  │
│  LLM推理 │             │   工具   │             │ ├─ 操作结果       │
│          │ ←────────── │          │             │ ├─ 当前状态       │
└──────────┘  下一步决策  └──────────┘             │ └─ 下一步线索     │
                                                   └──────────────────┘
```

工具在改变世界的同时，也在向 Agent 报告世界当前的样子。`exec` 运行测试套件，失败信息中的文件路径、行号、断言内容——都是直接定位到修改点的线索，而不只是"成功"或"失败"两种状态。这种信息密度的差异，直接决定了 Agent 在后续步骤中能否做出准确的诊断。

把工具调用的双重角色理解清楚，对设计工具返回格式有直接指导意义。一个被设计为"让人类读的"工具返回，会强调可读性、简洁性、摘要；一个被设计为"让 Agent 推理的"工具返回，会强调完整性、结构性、可追溯性。两者之间存在真实的张力。在构建 Agent 系统时，应该优先满足后者——Agent 的推理能力足以处理结构化的原始信息，但人类读者往往也可以直接查看工具调用历史，并不需要工具返回为他们做摘要。

好的工具设计不仅关心"如何做事"，也关心"如何汇报"。工具返回的是**证据**，结论由 Agent 的推理来得出。良好设计的工具返回包含三类信息：操作结果（做了什么、改变了什么）、状态信息（操作后相关部分的当前状态）、下一步线索（主动提示后续操作的方向）。要抵制"过度加工"的诱惑——把多个警告合并为"发现若干问题"、截断错误栈为最后几行，这些对人类读者友好，却悄悄削减了 Agent 可用的信息密度。

工具返回格式的设计，本质上是在回答一个问题：什么信息对 Agent 的下一步推理最有价值？这个问题的答案因任务不同而不同，但一个通用的原则是：保留结构，而非提炼摘要。`edit` 成功替换后，返回修改前后的对比，而不只是"替换成功"四个字；`exec` 运行构建时发现警告，即使构建成功，也应把警告完整返回——因为这些细节可能是下一个问题的根源。Agent 的推理能力足以处理大量原始信息，但它无法从被丢弃的信息中重建上下文。

### 正交性与错误作为信息

组合性的前提是工具的**正交性**：每个工具做自己的事，不与其他工具重叠，不产生令人意外的副作用。正交性保证工具可以自由组合，也让调试变得简单——当某个工具链出错，可以逐个隔离工具调用，找到哪个环节的返回触发了错误决策，而不是面对无法拆解的整体故障。

正交性也对工具测试产生深远影响。正交的工具可以被独立测试——测试 `read` 时不需要 `write`、`edit`、`exec` 参与，因为 `read` 不依赖于它们的状态。这种独立性让工具的行为易于验证，问题也易于定位。当四个工具的行为都经过充分测试和验证，它们在任意组合下的行为就是可预期的。系统层面的可靠性，从组件层面的清晰边界开始建立。

工具的正交性还有一个实践层面的好处：当新需求出现时，可以独立评估"是否需要新工具"。如果新需求可以通过现有工具的组合实现（即使组合步骤较多），通常不需要引入新工具；只有当新需求需要与现有工具根本不同的操作语义时，才有引入新工具的充分理由。这个评估框架帮助系统保持工具集的精简，避免随时间积累大量功能相似但边界模糊的工具。

成熟的工具系统把错误返回视为正常反馈的一部分。`edit` 的"找不到就报错"正是这种哲学的体现：错误信息传递了两层含义——文件当前内容与 Agent 预期不符，且文件可能已被此前操作修改。这两个推断直接指向下一步：重新 `read` 文件，再重新尝试。**工具的容错能力，一半体现在如何防止错误发生，另一半体现在如何清晰描述已经发生的错误。** 良好的错误信息是有导向性的——帮助 Agent 推断"问题成因是什么"，减少无效试探。工具是 Agent 的感知系统，它描述失败的方式与描述成功的方式同等重要。

这里有一个常见的反模式需要警惕：工具在遇到错误时"尽力而为"——部分完成操作然后返回成功，或者将错误包装成警告。这种设计对最终用户看起来更友好，但对 Agent 是危险的。Agent 依赖工具返回来建立对世界状态的准确认知，一个"部分成功"的返回会让 Agent 认为某件事已经完成，从而跳过后续的验证步骤，把一个未完成的中间状态带入后续操作。显式失败比隐式部分成功要好得多——前者让 Agent 有机会重新决策，后者让错误在系统中悄悄累积。

工具设计中的另一个重要考量是**幂等性**：同一个工具调用，在相同条件下执行多次，结果应该一致。`read` 天然是幂等的；`write` 在创建新文件时需要注意文件已存在的处理；`edit` 的精确匹配设计使其在找不到旧内容时报错，而不是静默地做重复替换，这是一种对幂等性的间接保护。幂等性对 Agent 系统格外重要——当 Agent 不确定某个操作是否已经执行（例如，任务被中断后重启），可以安全地重新执行幂等操作，而不用担心产生副作用的叠加。

## 五、技能——知识的模块化

### 工具与技能的根本区别

四个基础工具解决了"能做什么"，而**技能（Skills）**解决的是"怎么做得专业"——两个不同维度的问题。工具扩展执行能力，技能扩展专业知识。一个懂代码审查的 Agent 和不懂的 Agent，使用的工具集完全相同，区别在于它知不知道：应该优先检查安全漏洞，其次是性能问题，最后是代码风格；某类写法是反模式；某个框架有特定最佳实践。

这个区分揭示了一个重要的系统设计原则：**能力与知识是两个独立的扩展轴**。在工具体系之外单独维护知识体系，比把知识硬编码进工具实现或系统提示词里，有更好的灵活性和可维护性。知识可以独立迭代——随着对某个领域的理解加深，只需更新对应的技能文件；工具保持稳定——因为它们的职责是通用的执行能力，不依赖于领域知识的变化。两个扩展轴分离，意味着在任何一个轴上的变化，都不需要影响另一个轴上的内容。

这种分离的另一个实际好处是：技能可以在团队间共享和复用，而工具通常是系统内置的。一个在 Web 开发项目中积累的"前端性能优化"技能，可以被具有相同工具集的不同项目甚至不同团队直接采用，而不需要任何代码层面的改动。这让组织级别的知识积累变成了可以被系统性管理的资产，而不是分散在各个 Agent 实例配置里的碎片信息。

技能是一个 Markdown 文件，包含某领域的专业知识、检查清单、流程规范、决策指导。技能不增加新工具，而是告诉 Agent 如何更好地使用已有工具完成特定领域的任务。这个区分有工程上的实际意义：技能的更新不需要修改代码、不需要重新部署、不需要重启运行时——只需要编辑一个 Markdown 文件。知识与代码分离，使得专业知识的积累和维护变得极其轻量。

从认知角度看，技能解决的是"怎么用已有能力做得更好"，而不是"获得新的能力"。一个没有相关技能的 Agent 面对代码迁移任务，可能会在没有充分分析依赖关系的情况下开始改写，或者在测试失败后陷入低效的试错循环。一个加载了迁移技能的 Agent，则会在开始前系统地制定分析计划，在测试失败时按优先级处理问题，最终以更少的步骤完成同样的任务。工具集没有变化，变化的是决策质量。

技能与系统提示词（system prompt）的关系值得厘清。系统提示词里可以包含通用指导，但它对所有任务都是可见的，会占用宝贵的上下文空间。技能的按需加载机制解决了这个问题——只有当前任务需要的专业知识才会进入上下文。从架构角度看，技能是系统提示词的模块化延伸：系统提示词放的是 Agent 在所有场景下都需要的基础知识，技能文件放的是特定场景下才需要的专业知识。两者协同工作，共同构成 Agent 的完整知识体系，但各自占据最合适的生命周期。

### 按需加载——上下文的最优利用

技能不预先加载到系统提示词里，而是 Agent 判断需要时通过 `read` 读取其内容获取。这让上下文里只有"当前用到"的知识，最大化了有限窗口的利用率。技能文件更新后，下次 Agent 读取时自然获得最新版本——知识与运行时解耦，维护成本大大降低。

按需加载的触发机制本身是一个有趣的设计问题：Agent 如何知道"现在需要加载某个技能"？这依赖系统提示词中对技能体系的简要描述——告知 Agent 有哪些技能可用、分别适用于什么场景，但不把技能的完整内容都放进来。这是一种元知识的设计：Agent 知道"自己知道哪些可以知道的事"，并在需要时按需获取。这个设计模式在大规模知识系统中有普遍价值——完整性与效率之间的平衡，通过元信息层来实现。

不同于工具的静态注册（工具在系统启动时就已确定），技能的动态加载意味着 Agent 的知识范围在任务执行过程中可以扩展。一个任务从普通代码编写开始，然后遭遇性能问题——Agent 可以在这个时刻主动 `read` 性能优化技能，用刚获得的专业知识处理当前的子问题，然后继续原有任务。这种动态知识扩展能力，让技能体系不只是静态的参考手册，而是 Agent 在复杂任务中主动调用的专业顾问。

```
┌────────────────────────────────────────────────────┐
│  技能 vs 工具：两个扩展维度                         │
├─────────────────────┬──────────────────────────────┤
│  扩展工具            │  扩展技能                    │
├─────────────────────┼──────────────────────────────┤
│  增加 Agent 能做的事 │  增加 Agent 知道的事          │
│  通用性高            │  专域性强                    │
│  内置于运行时        │  按需从文件动态加载           │
│  改变执行能力        │  改变决策质量                 │
└─────────────────────┴──────────────────────────────┘
```

当你发现需要频繁给 Agent 补充某类指导才能把任务做对，说明这类指导具备被沉淀为技能的价值；当技能里的步骤总是离不开某个重复的工具调用序列，这个序列可能值得被包装为更高阶的专用工具。工具和技能之间的边界随实践经验不断演化，这是系统成熟度的自然表现，不是设计缺陷。工具负责稳定的执行能力，技能负责演化的领域知识，两者解耦正是这个架构持续可维护的原因之一。

技能的另一个价值在于它是可以被版本化和共享的。一个团队积累的"Python 重构最佳实践"技能，可以直接被另一个使用相同工具集的团队或项目复用，而无需修改任何代码。这让组织内的 Agent 能力积累变成了真正意义上的知识资产——不依附于特定的运行时，不需要特殊的基础设施，只是一些经过验证的 Markdown 文件。技能体系因此也成为了团队内显性化和传播领域知识的一种实用机制。

### 可选工具扩展

当任务需要超出本地文件系统和命令行的能力范围时，可以引入可选工具作为扩展补充：

| 工具 | 用途 | 适用场景 |
|------|------|---------|
| `web_search` | 搜索网络，获取实时信息 | 查找文档、调研技术方案 |
| `web_fetch` | 抓取指定网页的内容 | 读取在线文档、获取 API 响应 |
| `browser` | 控制浏览器：点击、填写表单、截图 | 自动化测试、爬取动态内容 |
| `apply_patch` | 应用标准 diff 格式的补丁文件 | 批量代码变更、补丁管理 |

这些工具说明四个基础工具的"最小完备集"面向的是本地文件系统和命令行环境。核心架构保持稳定，扩展点明确——四个原语提供确定性基础，可选工具在需要时扩充边界。

引入可选工具时，工具描述的质量同样至关重要。`web_search` 和 `web_fetch` 的能力存在重叠——都能获取网络内容；描述必须清楚说明两者的使用边界：前者用于发现信息（不知道具体 URL 时），后者用于获取已知地址的内容（知道 URL 时）。这个描述边界的清晰与否，直接决定 LLM 在实际使用时是否会产生选择混乱。可选工具扩展了能力边界，但同时也扩大了需要管理的描述复杂度，这是两者之间不可回避的权衡。

## 本章小结

本章的核心洞见是：**四个原语工具——每个只做一件事——通过组合创造出无限能力。秘密不在工具本身，而在于每次工具调用都将观察结果反馈回推理循环，使工具系统同时成为 Agent 的双手和感知系统。** 单个工具的简单性不是局限，而是设计意图——简单性保证了正交性，正交性保证了组合性，组合性才是真正的超能力。四个工具加技能的最小完备集，在上下文窗口约束下实现了能力与清晰度的最优平衡。

理解这一洞见，需要抵制一个直觉上很合理但在工程上有误的倾向——把工具理解为"功能更丰富更好"。功能的丰富性在工具层面并不直接转化为任务的成功率，因为中间还有一个关键环节：LLM 的工具选择决策。工具设计最终服务于的不是"什么技术上能实现"，而是"LLM 能否在对的时刻选择对的工具，并从工具返回中提取出有效信息推动任务进展"。这个框架下，简单、正交、描述清晰的工具体系，比功能丰富但边界模糊的工具集，在系统层面表现更好。

这一架构的意义超越功能层面。工具是 Agent 与真实世界的接触面，其设计质量直接决定 Agent 行为的可预期性。`read` 的三层防御保护上下文完整性，`write` 的原子写入保护文件完整性，`edit` 的精确匹配防止静默误改，`exec` 的 PTY 支持确保与外部系统的真实交互——每一处设计都在回答同一个问题：如何让 Agent 在行动时既有力量，又有边界。工具的描述是写给 LLM 的 API 文档，名称与意图的对齐是让 LLM 在没读完描述之前就能做出正确预判的工程基础。工具名称、描述、参数设计共同构成了工具的"接口合同"，这份合同的质量直接决定 LLM 在工具边界模糊地带的选择准确率。

从另一个角度看，工具系统定义了 Agent 可以改变世界的方式。不同的工具设计，会塑造出不同的 Agent 行为模式——权力集中的少量强工具，倾向于产生步骤少但风险高的操作序列；权力分散的多个精细工具，倾向于产生步骤多但每步可控的操作序列。OpenClaw 选择的是后者：每个工具做一件明确的事，操作序列更长，但每一步的意图更清晰、风险更可控、错误更容易被识别和纠正。这不只是技术选择，也是关于 Agent 应该如何与世界交互的设计哲学。

技能在此架构之上引入了第二个扩展维度——不增加能力，而扩展知识。四个工具加上按需加载的技能，构成了一个在上下文窗口约束下兼顾通用性与专业性的完整系统。工具设计的终极目标不是让 Agent 能做更多，而是让 Agent 在每一步都能做对：有清晰的感知、准确的执行、充分的反馈、安全的边界。当这四件事同时做好，复杂任务的完成就是这些简单原则自然叠加的结果。

工具系统的质量，最终体现在 Agent 处理意外情况时的鲁棒性。预期路径上的成功是必要条件，不是充分条件。一个设计良好的工具系统，应该让 Agent 在遇到文件被意外删除、命令超时、内容与预期不符等情况时，能够通过工具返回的信息迅速理解发生了什么，并基于这个理解做出合理的下一步决策——而不是陷入混乱、重复无效的操作。工具的返回是 Agent 在异常情况下的主要信息来源；当这个来源的信息质量足够高，Agent 的自我纠错能力就自然地从中涌现出来。

**关键要点：**

- 四个原语工具覆盖感知-推理-行动的完整闭环，通过组合产生无限能力；工具数量超过临界点后边界模糊导致选择失误率上升
- `read` 三层防御保护上下文；`write` 原子性写入保护文件完整性；`edit` 精确匹配防止误改；`exec` 连接外部世界，PTY 支持确保真实终端交互
- LLM 通过描述文本理解工具，描述质量直接决定选择准确率；工具名称与自然语言概念对齐是意图对齐的第一道设计
- 每次工具调用的返回值是 ReAct 循环的新一轮观察；错误返回是导向下一步的信息，不是需要掩盖的意外；工具返回的是证据，结论由推理得出
- 技能扩展领域知识，工具扩展执行能力，两个维度分别演化；技能的按需加载机制使知识与运行时解耦，维护成本极低
- 工具设计的正交性保证独立可测试性；幂等性保证中断后可安全重试；显式失败比隐式部分成功更有利于 Agent 的自我纠错
- 工具描述是随系统演进而演进的接口合同，需要像接口文档一样定期审查
- 组合性让 Agent 通过工具链实现工具集中不直接存在的操作——不需要为每一类操作预先准备专用工具
