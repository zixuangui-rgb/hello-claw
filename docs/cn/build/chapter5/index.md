# 第五章 消息循环与事件驱动——Agent的"心跳"

> **核心问题**：一条消息从发出到被处理，经历了哪些环节？多个会话同时活跃时如何保证互不干扰？Agent如何主动"醒来"做事？拥有时间感知的Agent，与无状态的API调用，究竟有何本质不同？

本章回答这些问题的主线是两个设计决策：**泳道模型**（让Agent在并发环境下可靠运行）和**心跳机制**（让Agent在时间维度上拥有主动性）。两者共同奠定了Agent作为持续运行实体的基础架构。

理解这两个机制，也就理解了为什么Agent系统需要专用运行时，而不能只是"加了工具调用能力的API"。并发安全和时间主动性都是运行时层面的能力，它们要求对消息流、调度、状态持久化进行统一管理——这些远超单次函数调用所能承载的范围。

本章按如下顺序展开：首先建立消息调度的基础（命令队列与泳道模型），然后引入任务分类体系，再进入时间感知的三个层次，继而深入心跳机制的设计哲学，最后以分层容错收尾。每一节都服务于同一个主题：**使Agent从一次性调用进化为持续存在的智能体**。

## 一、命令队列与泳道模型

### 为什么不能"收到就执行"

"收到消息就交给Agent处理"看似直接，但在工程实践中会立即遇到三个问题：

| 问题 | 后果 |
|------|------|
| 并发冲突 | 两条消息同时修改同一文件，数据损坏 |
| 无缓冲 | 高峰期消息涌入，系统过载 |
| 无序执行 | "创建文件"和"读取文件"乱序，后者读不到前者的结果 |

这三个问题共同指向一个根本矛盾：**Agent是有状态的，但消息的到达是无序且并发的**。解决这个矛盾，需要在消息到达和消息执行之间插入一个调度层——这就是**命令队列（Command Queue）**的作用。命令队列不是性能优化，它是正确性保证：没有命令队列，Agent的状态在并发压力下随时可能进入不一致状态。

可以用一个类比来理解：厨师不会因为同时收到十张订单就同时开始烹饪所有菜品——他会将订单排队，一道一道按序出餐。命令队列的作用与此相同：它将无序到达的消息转换为有序的执行序列，让Agent能够在一个确定性的环境中工作。队列的存在使Agent的状态变化变得可追踪、可审计——每一次状态变化都对应队列中的一条消息，消息的顺序就是状态变化的历史。

### 事件驱动 vs 请求-响应

请求-响应模式下，调用方发出请求并等待结果。这对执行时间可预期的场景没有问题，但Agent的执行时间不可预测（一个问题可能三秒回答，也可能需要五分钟），且需要同时处理来自用户、定时器、子Agent的多源消息——同步等待会立即导致死锁。

| 维度 | 请求-响应 | 事件驱动 |
|------|---------|---------|
| 发送者行为 | 发出后等待结果 | 发出后继续执行 |
| 时间耦合 | 发送者与执行者同步 | 两者在时间上解耦 |
| 主动触发 | 无位置放置"无人发起的消息" | 心跳/定时任务天然融入 |
| 适合场景 | 执行时间短、单源调用 | 执行时间不定、多源并发 |

命令队列是事件驱动思想的实现载体：消息先进队列，再由调度器有序取出执行。这个模型的另一个好处是**消息来源的统一化**：用户消息、心跳触发、Cron到期、子Agent回调，在架构上都是"一条消息进入队列"，调度器不区分它们的来源，只负责有序执行。这种统一化使得系统的扩展变得简单——新增一种触发源，只需让它将消息投入队列，无需修改调度逻辑。统一的消息模型也简化了监控：观察队列就能看到系统全貌，而无需分别监控不同来源的事件流。

### 泳道模型——顺序与并发的优雅平衡

命令队列的核心调度策略叫做**泳道模型（Swim-Lane Model）**，规则只有两条：

- **同一泳道内**：消息**严格串行**（一个处理完再处理下一个）
- **不同泳道间**：消息**并行**处理（互不等待）

**一个泳道 = 一个会话（Session）**

```
泳道A（会话A）：消息1 ──────────────────→ 消息2 ──→ 消息3
                           顺序执行，严格串行

泳道B（会话B）：          消息4 ──→ 消息5
                                     与A完全并行，互不影响

泳道C（会话C）：                消息6 ──────────────────→ 消息7
```

泳道模型的本质是**用空间换秩序**：以"会话"为隔离单元，会话内保证顺序，会话间保证并行。这与Actor模型（Erlang/Akka的理论基础）高度相似——每个Actor有自己的消息信箱，串行处理但多个Actor并行运行——不同之处在于泳道模型将"会话"这个业务概念直接映射到了并发控制结构上。这使得隔离粒度与用户感知的业务边界自然对齐，无需人工维护额外的并发锁。

泳道隔离还带来了一个实际工程优势：**单个泳道的故障不污染其他泳道**。如果某个会话内的任务进入了死循环或异常状态，调度器可以终止该泳道而不影响其他会话的正常运行。这与操作系统进程隔离的思想一脉相承——隔离是可靠性的基础。

会话内串行的核心理由是**因果一致性**：用户发出"把.log文件压缩为archive.zip"再发出"上传archive.zip"，第二条消息隐含了第一条已完成的假设。串行执行，是将这种人类常识编码进架构的方式。因果一致性不是可选的性能特性，而是使Agent输出可预期的基础前提。

值得注意的是，因果一致性只需要**在同一会话内**保证，而非跨越所有会话。这个范围设定是精心设计的：用户与Agent的对话形成了一个因果链（每条消息都假设了前面消息已被处理），而不同用户的会话之间不存在这种因果关系。泳道模型的隔离粒度与因果依赖的实际范围完全吻合，避免了过度串行化（将所有会话排成一队）和不足串行化（同一会话内允许并发）两种极端。

### 消息插入的几种模式

Agent正在执行长任务时，新消息的处理策略取决于用户意图：

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| `collect` | 等Agent完成后，将等待中的多条消息合并跟进 | 大多数情况（默认） |
| `steer` | 立即注入当前运行，引导Agent调整方向 | "不对，先做X再做Y" |
| `interrupt` | 立即中断当前运行，处理新消息 | 紧急停止或方向性错误 |
| `followup` | 排队到当前任务完成后处理 | 有明确先后顺序的任务序列 |

消息调度不是技术问题，而是**用户意图的解读问题**。同一条消息"先做X"，在任务刚开始时可能是 `steer`（还来得及调整），在任务接近完成时可能是 `followup`（等完成后再安排X）。调度策略需要与用户意图对齐，而非机械地套用固定规则。

不同的插入模式也对应不同的用户心态：
- `collect` 假设用户可以等待、信任Agent完成当前任务
- `interrupt` 假设用户认为当前任务方向已经错误、需要立即纠正
- `steer` 假设用户希望在不丢失当前进度的情况下微调方向

将这些心态显式建模为调度参数，是将用户控制权具体化的工程方式。用户不需要了解队列的实现，只需要选择自己意图最匹配的模式。

## 二、四种任务类型

| 类型 | 触发方式 | 典型场景 |
|------|---------|---------|
| **Main**（主任务）| 用户发送消息 | 日常对话、指令执行 |
| **Cron**（定时任务）| 时间触发 | 每日报告、定期同步 |
| **Subagent**（子代理）| 父Agent调用 | 大任务分解，委托子Agent执行 |
| **Nested**（嵌套任务）| 任务执行中产生 | 主任务发现需要先完成前置任务 |

每种任务类型的关键属性：
- **Main**：低延迟，在主会话泳道中串行执行，直接响应用户
- **Cron**：准时性，在独立泳道触发，可有或无上下文历史
- **Subagent**：并行性，多个子代理实例可同时运行于各自泳道
- **Nested**：依赖性，共享父任务的上下文，完成后将控制权归还父任务

任务分类的意义不只是"谁触发了它"，更在于**明确调度权**：Main任务需要低延迟响应；Cron任务需要准时触发但不紧急；Subagent任务可以并行；Nested任务依赖触发它的任务的上下文。不同调度语义意味着不同的优先级、隔离级别和资源分配策略。

从另一个角度看，任务分类也是一种**意图声明**。当系统知道某个任务是Cron任务时，它可以预先分配资源、在触发时刻提前准备执行环境；当系统知道某个任务是Subagent任务时，它知道可以为多个实例分配独立资源而无需顺序排队。任务类型是Agent系统与调度层之间的契约。

分类带来的最直接好处是**调度隔离**：

```
时间轴 ──────────────────────────────────────→

Main（用户对话）：  [任务A: 5分钟代码审查]
Cron（定时任务）：                    [9:00日报]
Subagent（子代理）：  [子任务1]  [子任务2]  [子任务3]
```

5分钟的代码审查不会推迟早9点的日报，三个子任务并行执行而非顺序等待。这种隔离在单线程调度模型中是不可能的——没有任务类型的概念，长任务天然会阻塞所有后续任务。任务分类将调度从"先来先到"升级为"按类型分配资源"。这一升级对用户的实际感知影响极大：用户期待对话回复的低延迟（Main任务），但不会在意日报晚三秒生成（Cron任务）；系统能够区分并分别满足这两种不同的延迟期望。

Nested任务是四类中最微妙的：它由主任务在运行中动态产生，继承父任务的上下文，但作为独立任务单元管理。当主任务发现"要完成目标A，必须先完成前置条件B"时，它将B创建为Nested任务，等B完成后再继续。这是Agent系统中实现动态任务分解而不失去上下文的关键机制。

Nested任务与Subagent任务的区别在于**上下文继承**：Subagent运行在独立隔离的上下文中，适合无状态的并行子任务；Nested任务继承父任务的状态和上下文，适合"临时发现的前置依赖"。选择哪种取决于子任务是否需要知道"它是在哪个大任务中被调用的"。

## 三、时间感知——Agent的第四维度

传统API调用是无时间感知的——函数不知道上一次被调用是什么时候，不会主动在某个时刻"醒来"。每次调用都是独立的、无记忆的。对助手型Agent而言，这是根本性的残缺：一个不知道时间的助手，无法感知"用户已经两天没有回复这个紧急问题"，也无法主动在约定时刻执行检查。更深层的问题是：无时间感知的Agent只能响应，无法主动——它的所有行为都由外部触发决定，它自身没有节律。Agent的时间感知分为三个层次：

```
第一层：知道现在几点
  └─ 在提示词中注入当前时间戳，Agent能做时间相关判断

      ↓

第二层：知道距离上次交互过去了多久
  └─ 持久化会话状态，记录最后活跃时间，感知"沉默的时长"

      ↓

第三层：主动在某个时刻触发自身  ← 心跳机制实现的层次
  └─ Agent能设置"闹钟"，在约定时间点自主执行检查清单
```

大多数Agent框架止步于第一层，少数做到第二层。心跳机制实现的是第三层——**时间是Agent从工具进化为助手的关键维度**。第一层让Agent能回答"现在几点了"；第二层让Agent能感知"我们已经很久没有联系"；第三层让Agent能主动说"到时间了，让我检查一下"。每一层都是前一层的质变，而非量变。

从实现角度看，三层时间感知的技术要求也逐层递增。第一层只需在提示词构建时注入系统时间，任何框架都能轻松实现。第二层需要持久化会话状态，跨调用保存最后活跃时间戳，涉及存储层的设计。第三层需要独立于用户请求运行的调度进程，能够在没有外部触发的情况下主动唤醒Agent——这是架构上的根本性跨越，不是在现有框架上添加一个功能，而是改变了Agent的运行模型。

三层时间感知的区分对评估一个Agent框架的成熟度很有实用价值：
- 框架支持**第一层**：可构建对话助手，能回答时间相关问题
- 框架支持**第二层**：可构建有记忆的持续助手，感知用户沉默时长
- 框架支持**第三层**：可构建真正自主运行的Agent，自行安排周期性任务

在评估或选择框架时，时间感知层次是一个简单而有区分度的维度。

拥有时间感知后，Agent具备了完整的生命周期，不再是"调用时存在、结束则消失"的无状态函数：

```
  [启动]
     |
     ↓
  [等待] ←─────────────────────────────────┐
     |                                      |
     |── 收到用户消息 ──→ [执行Main任务] ──→ [完成，回到等待]
     |                                      |
     |── 心跳触发 ──────→ [执行检查清单] ──→ [完成，回到等待]
     |                                      |
     └── Cron触发 ──────→ [执行定时任务] ──→ [完成，回到等待]
```

Agent在没有触发时处于休眠，但这是**主动的休眠**——它知道下一次心跳和Cron任务什么时候执行，它的存在不依赖于某次特定的调用。这个生命周期图揭示了一个重要的架构事实：等待本身是Agent状态的一部分，而非"Agent不存在"的状态。

这个持续性是区分Agent与传统API的深层标志。API端点的生命周期是"请求到达→处理→响应→消亡"，它在请求之间不存在。而有生命周期的Agent在"等待"状态时依然存在：它有记忆（会话历史）、有计划（已安排的Cron和心跳）、有身份（用户与它建立的关系）。这种持续性使得Agent能够建立跨时间的上下文理解，而不是每次对话都从零开始。

## 四、心跳机制与主动性哲学

心跳机制是本章最具原创性的设计。在大多数Agent框架中，主动性是一个待解决的问题而非已有答案的实现。心跳机制给出了一个具体的工程化路径：以固定间隔唤醒Agent执行检查清单，以约定的静默信号过滤无意义输出。这三个要素——定期唤醒、用户定义清单、HEARTBEAT_OK 静默——共同构成了一个最小可用的主动性架构。

### HEARTBEAT.md：检查清单

一个只在你问时才说话的管家，无法告知你：重要邮件三小时前未处理、监控服务已停止响应、后台任务等待指令。心跳机制让Agent定期"醒来"检查——有则通知，无则沉默。

心跳机制刻意保持简单：在 `HEARTBEAT.md` 里用自然语言写检查清单，Agent被定期唤醒时逐项执行。**检查清单由用户定义，而非系统硬编码**——个性化不需要代码，只需要编辑一个文本文件。这个设计决策反映了一种哲学：机制（定时唤醒）由系统提供，但内容（检查什么）由用户决定。将两者分离，使得心跳机制可以服务于任何使用场景，而无需预先知道用户的具体需求。

这个分离也有一个重要的可维护性优势：随着用户需求变化，他们只需修改 `HEARTBEAT.md` 中的自然语言清单，而不需要修改任何代码、重新部署任何服务。文本文件是最低摩擦的配置方式——它可以被版本控制、被团队共享、被随时调整，而心跳的触发机制保持不变。

### HEARTBEAT_OK：静默哲学

心跳机制最精妙的设计，不是如何触发，而是如何**不打扰**。如果每次心跳都推送消息，哪怕内容只是"一切正常"，机制也会很快被关掉——它制造了噪音。通知疲劳是主动性设计失败的最常见原因：系统变得"过于健谈"，用户开始忽略所有通知，真正重要的信息也因此被淹没。

解决方案是一个约定：检查后无需告知时，Agent只回复 `HEARTBEAT_OK`，系统**静默丢弃**，用户无感知。

```
心跳触发
  ↓
Agent读取 HEARTBEAT.md，逐项检查
  ↓
  ├── 没有需要关注的事 → 回复 HEARTBEAT_OK → 系统丢弃，用户无感知
  │
  └── 发现需要关注的事 → 回复实际内容 → 用户收到通知
```

这体现了一个更普遍的原则：**主动性的价值，在于它只在有意义时发声**。无效心跳零打扰，有内容才通知。`HEARTBEAT_OK` 是一个协议约定，而非实现细节——它将"静默"从默认行为变成了明确的架构选择。

从系统设计角度，`HEARTBEAT_OK` 的静默丢弃机制也带来了一个实用好处：它使心跳频率可以设置得相对高一些，而不必担心"过于频繁的通知"问题。在没有异常情况时，高频心跳完全无感；一旦出现需要关注的情况，响应延迟也更低。这是一个以计算资源换取响应灵敏度的合理权衡，而 `HEARTBEAT_OK` 机制使这个权衡的用户侧成本降为零。

### 何时主动，何时等待

主动性的边界取决于**信息的时效性**和**用户的注意力成本**：

```
高时效性 + 低注意力成本 → 应该主动
  例：服务宕机告警。每延迟一分钟损失增加，推送通知成本极低。

低时效性 + 高注意力成本 → 应该等待
  例："今天读了篇好文章"。不紧急，打断工作流有真实认知成本。

不确定时 → 默认等待
  HEARTBEAT_OK 的静默设计，本质上是把"默认等待"编码进架构。
```

好的主动Agent还需要自我抑制：同一问题已通知则不重复推送；用户正在对话时，非紧急心跳结果排队等候。主动性的工程化，不是"让Agent更努力推送"，而是"让Agent更聪明地判断何时不说话"。

主动性与被动性并非二元对立，而是一个需要动态校准的参数。对于同一个用户，在深度工作时段应当偏向被动（只在紧急情况打断），在空闲时段可以偏向主动（分享上下文相关的信息）。理想的Agent不是固定在某个点上，而是能读取上下文、动态调整自己的主动程度。这是心跳机制从硬编码触发频率进化为用户可配置清单的深层原因。

心跳与Cron在用途上有明确分工：

| 对比维度 | 心跳（Heartbeat） | Cron定时任务 |
|---------|----------------|------------|
| 触发模式 | 固定间隔周期触发 | 标准时间表达式，精确到特定时刻 |
| 执行上下文 | 主会话，有完整对话上下文 | 可选主会话或独立隔离会话 |
| 任务性质 | 周期性监控、持续状态检查 | 精确时间点的确定性任务 |
| 静默机制 | HEARTBEAT_OK，无结果时丢弃 | 可配置结果是否推送 |

选择建议：需要在精确时刻执行（"每周一发送周报"）→ 用Cron；需要持续关注状态（"随时检查紧急消息"）→ 用心跳；两者可以组合，分别处理各自最擅长的场景。

实际部署中，很多场景同时需要两者。例如：用心跳每15分钟检查是否有紧急消息需要处理（持续监控），同时用Cron在每天下午5点生成工作日报（精确时刻任务）。两者互补，覆盖了主动性的两种基本形态：**持续感知**（心跳）和**定时执行**（Cron）。

两者的另一个区别在于**上下文归属**。心跳在主会话中执行，Agent能看到完整的对话历史，因此心跳执行的检查可以参考过去的对话内容（比如"用户上次提到要追踪这个任务"）。Cron任务可以选择独立会话执行，与任何特定用户对话解耦——更适合面向系统的任务（生成报告、清理缓存）而非面向用户的提醒。理解这一区别，有助于在设计时为每类任务选择合适的触发机制。

## 五、容错与自愈

### 分层容错

对长期运行的Agent来说，"系统一直完美运行"不是正常状态，而是一种幻觉。外部API会超时，文件系统会权限不足，LLM服务会偶发性不可用。容错设计的出发点不是"如何避免故障"，而是"故障发生时，如何把影响最小化"。这种思维转变本身就是架构成熟度的标志：从"假设成功、例外处理失败"转向"假设失败会发生、预先设计隔离边界"。

Agent的执行路径涉及文件系统、外部API、LLM服务等多个依赖，每个环节以不同方式失败，需要不同处理策略。分层容错的逻辑是**在距离故障最近的地方处理它**：

```
┌─────────────────────────────────────────────┐
│  第3层：服务级容错                            │
│  LLM服务不可用 → 切换备用模型 → 降级处理       │
│                                              │
│  ┌───────────────────────────────────────┐   │
│  │  第2层：会话级容错                     │   │
│  │  任务整体失败 → 隔离影响 → 会话保持活跃  │   │
│  │                                       │   │
│  │  ┌─────────────────────────────────┐  │   │
│  │  │  第1层：任务级容错               │  │   │
│  │  │  工具调用失败 → 重试 / 换策略    │  │   │
│  │  └─────────────────────────────────┘  │   │
│  └───────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
```

三层容错各司其职，互不越权。工具级失败不应上升为会话级终止，会话级失败不应导致整个服务不可用。这种隔离的本质是**故障影响范围的精确控制**：每一层都在自己能力范围内处理故障，超出能力才上报上一层。

值得注意的是，三层容错之间存在一个重要的信息流方向：故障从内层向外层上报时，外层拥有更大的修复权限——任务级容错可以重试工具调用，会话级容错可以中止整个任务并恢复会话，服务级容错可以切换底层模型。权限随层级升高而扩大，但激活频率随层级升高而降低。正常运行的系统中，第1层容错频繁触发（网络抖动是常态），第2层偶尔触发（任务逻辑出错），第3层极少触发（服务级故障）。

容错设计的一个常见误区是过度在第1层处理本该上报的错误。工具调用失败时，有时最好的处理不是重试，而是将错误上报给推理层——让Agent决定是否换一种方式完成目标。过度重试会掩盖真实问题，延迟用户获得有用反馈的时间。第1层应当处理明确的瞬时错误（网络抖动），对于语义性失败（"文件不存在"意味着计划需要修正），应当快速将控制权还给推理层。

### 错误即观察

Agent容错的核心理念是把错误视为**信息而非终止信号**。在ReAct（推理-行动）循环中，工具调用的结果——无论成功还是失败——都作为"观察"反馈给推理步骤：

```
Agent计划：读取 /tmp/config.json
  ↓ 执行 → 文件不存在（错误 = 观察）
  ↓ 推理：文件不在预期位置，需要先找到它
  ↓ 行动：搜索项目目录中的 config.json
  ↓ 找到 ./src/config.json → 任务继续
```

错误修正了Agent对现实的认知，推动了更准确的下一步行动。这与传统程序的异常处理有根本不同：传统程序遇到异常要么重试要么终止，而Agent将错误纳入推理循环，可以基于错误信息动态调整执行路径。局部故障不扩散为全局故障：任务内的工具失败不影响会话存活，一个会话失败不影响其他泳道，LLM服务不可用通过服务级容错降级处理。

"错误即观察"这个理念有一个重要的实践含义：Agent在设计工具调用时，应当**主动返回有信息量的错误**，而非简单的"失败"。"文件不存在"比"操作失败"包含更多推理线索；"权限被拒绝"比"无法读取"更能帮助Agent判断下一步应该做什么。错误的信息质量直接影响Agent的自愈能力。

### 哪些错误该重试

| 错误类型 | 特征 | 处理方式 |
|---------|------|---------|
| 临时网络问题 | 连接超时、偶发断开 | 指数退避后重试 |
| 频率限制 | 服务返回"请求过多" | 等待后重试 |
| 认证失败 | API密钥无效或过期 | 不重试，需修改配置 |
| 资源不存在 | 文件路径错误、记录已删除 | 不重试，Agent重新规划 |
| 逻辑错误 | 输入不符合业务规则 | 不重试，理解原因后换策略 |

**指数退避**是行业标准：每次重试前等待更长时间，防止大量客户端同时重试雪上加霜。判断是否重试的核心问题是：**等待之后，情况会改变吗？** 网络超时等待后情况可能改善；认证失败等待多久情况都不会改变——只有修改密钥才能解决。

重试策略的另一个维度是**最大重试次数**。无限重试在某些情况下会消耗大量资源而无法带来结果——当外部服务长期不可用时，持续重试只会增加负载。合理的重试策略包含次数上限和最终的降级行为：超过上限后，任务标记为失败，错误作为观察反馈给上层推理，由Agent决定是通知用户、切换备选方案还是中止整个工作流。容错的终极目标不是"永不失败"，而是"失败时知道如何优雅地告知用户"。

## 本章小结

本章的核心洞察是：泳道模型与心跳机制共同赋予了Agent传统软件所不具备的两种属性——**会话内有保障的因果顺序**，以及**时间驱动的自主主动性**。两者合力，将Agent从一个无状态函数转变为持续运行、自我管理、拥有自身节律的实体。泳道模型不只是并发控制技术，它将业务概念（会话）直接编码为调度单元；心跳机制不只是定时任务，它赋予Agent第三层时间感知，使"主动"成为架构能力而非个案实现。

这一转变的意义在于：因果顺序解决了"Agent能否可靠地做事"的问题，而时间主动性解决了"Agent能否在正确的时刻做事"的问题。前者是基础正确性保证，后者是助手价值的来源。没有泳道模型，并发状态下的Agent是不可信赖的；没有心跳机制，Agent只是一个被动响应的工具，而非真正意义上的助手。HEARTBEAT_OK 的静默设计则说明：主动性的核心不是"更频繁地打扰用户"，而是"只在有意义时发声"——克制本身也是能力，是使主动性机制能够长期存活的工程智慧。

这些机制合在一起，也回答了一个更深层的问题：**为什么Agent需要专门的运行时，而不能只是一个被调用的函数？** 因果顺序需要队列，时间主动性需要调度进程，生命周期需要持久化状态，容错需要隔离层次——这些都是运行时层面的能力，无法通过单纯的函数调用实现。Agent运行时不是语言模型的包装层，而是使语言模型能够作为持续运行的智能体工作的基础设施。

- 命令队列是正确性保证，也是异步解耦的基础：消息先进队列再执行，事件驱动让发送者与执行者在时间上分离
- 泳道模型将并发安全与业务语义对齐：会话内串行保证因果一致性，会话间并行保证吞吐量，隔离粒度与业务概念天然匹配
- 四种任务类型使调度隔离成为可能：Main/Cron/Subagent/Nested各有独立调度语义，长任务不拖累实时响应
- 时间感知是从工具到助手的关键跨越：心跳机制赋予Agent第三层时间感知——主动在约定时刻触发自身，Agent成为有生命周期的持续运行体
- HEARTBEAT_OK 体现主动性的克制：无效心跳零打扰，有内容才通知，"默认等待"编码进架构，从根本上防止通知疲劳
- 分层容错将失败隔离在最小范围：错误作为观察纳入ReAct决策循环，这是"拥抱故障"而非"避免故障"的架构哲学
- 任务类型分类是调度语义的契约：它不仅描述触发方式，更决定优先级、上下文归属和隔离边界
- 消息插入模式是用户意图的工程化表达：四种模式对应四种不同的控制需求，让用户无需了解队列即可精确表达意图
- Agent运行时不是语言模型的包装层：它是使语言模型能够可靠、持续、自主运行的基础设施
