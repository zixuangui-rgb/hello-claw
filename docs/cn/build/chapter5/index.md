# 第五章 消息循环与事件驱动——Agent 为什么能稳稳地接住每一条消息

> **核心问题**：消息会乱序到达、任务会并发发生，OpenClaw 到底靠什么保证“不断线、不串线、还能主动工作”？

---

上一章讲的是工具系统。工具解决的是“Agent 终于能动手了”；这一章接着解决另一个问题：**动手以后，系统怎么不乱。**

因为只要 Agent 真开始做事，就一定会遇到这些情况：消息几乎同时到达，任务跑到一半被改方向，长任务还没结束又来了提醒或定时任务，没人说话时系统还得定期自查。表面上这是“消息处理”，本质上是在讲：**一个会持续协作的 Agent，怎样维持自己的运行节律。**

所以这一章其实在讲两套基础设施：消息循环负责让消息有序流动，事件驱动负责让系统在合适的时候自己触发动作。前者解决乱，后者解决僵。

---

## 第一节 为什么 OpenClaw 需要消息循环与事件驱动

这一节只回答一个问题：**为什么 Agent 不能继续沿用普通聊天系统那套“收到消息就回一条”的做法。**

### 1.1 从“一问一答”到“持续协作”，问题已经变了

普通聊天系统最自然的工作方式，其实很简单：

```text
收到一句话
→ 生成一段回复
→ 结束
```

这个模式在问答场景下非常顺手。你问一个概念，它解释；你贴一段代码，它分析。它的任务边界很清楚：**把当前这句回好。**

但 OpenClaw 想做的不是这个。OpenClaw 想做的是：

```text
接住一个任务
→ 自己判断下一步
→ 需要时调用工具
→ 看结果再继续推进
→ 直到完成、卡住，或者需要你拍板
```

这时候，系统面对的就不再是“回复一句话”，而是“维护一个正在进行的现场”。而只要任务在进行，消息就不可能永远老老实实按顺序来。

一个很常见的场景就是：你发消息“先把测试跑一下”，Agent 刚开始执行，你又补了一句“等等，先切到 `feature/login` 分支”，同时一个定时巡检又到了触发时间。如果系统还是“谁先到就先处理谁”，现场马上就会乱。

所以问题已经不是“模型会不会回答”，而是：**系统能不能在事情连续发生时，守住顺序、状态和上下文。** 这就是消息循环存在的起点。

### 1.2 如果没有队列和隔离，会发生什么混乱

很多初学者第一次看到“队列”“lane”“session 隔离”这些词，会觉得它们像很重的工程术语。其实它们解决的问题非常日常。

可以先看几个最直观的例子：

| 场景 | 没有调度时可能发生什么 |
|------|------------------------|
| “先创建文件，再写入内容” | 写入先跑，结果提示文件不存在 |
| 两个人同时使用同一个 Agent | 上下文混在一起，回复串线 |
| 一个长任务跑了五分钟 | 后来的消息一直卡住，系统像死了一样 |
| 任务进行中用户改方向 | 系统没接住，继续沿着旧方向跑 |

这些问题背后其实都指向同一件事：**Agent 是有状态的。** 它不是每次都从零开始，而是带着“现在做到哪一步、桌上已经有什么材料”继续往下走。所以消息一乱，坏掉的就不只是当前这一句，而是整段任务现场。

于是 OpenClaw 不能把消息处理理解成“收件箱里有新消息，我就马上答”，而必须先解决一个更基础的问题：**消息到达以后，先整理，再执行。** 这个“先整理”的过程，就是队列；这个“别互相打扰”的机制，就是隔离。

### 1.3 为什么它必须是事件驱动，而不是简单请求-响应

再往前走一步，你会发现 OpenClaw 面对的根本不只是“用户消息”。它还要处理很多“系统自己冒出来的事”。

比如：

- 用户发来一条新消息；
- heartbeat 触发了一次巡检；
- 一个 cron 定时任务到点了；
- 某个子任务完成后回传了结果；
- 某次运行状态变化，系统需要继续跟进。

这些东西来源不同，但本质很像：**有一件新事情发生了，系统要决定要不要处理、什么时候处理、在哪个现场里处理。** 这就是事件驱动的视角。它不再把系统理解成“有人请求，我就同步回一个结果”，而是理解成：

```text
外部或内部发生一个事件
→ 事件进入统一入口
→ 调度器决定它该去哪
→ 系统按规则推进
```

这样做的好处有两个：入口统一，用户消息、heartbeat 和 cron 都能走同一套处理框架；时间解耦，发起者不用管系统此刻忙不忙，调度层自己决定何时执行、是否插队、是否延后。

这就是为什么 OpenClaw 不是简单地“收消息再回答”，而是要做成**事件驱动的消息循环系统**。如果把第一节收成一句话，就是：**当 Agent 的目标从“回一句话”变成“持续协作”，系统就必须学会先整理消息、隔离现场，再决定如何推进。**

## 第二节 消息循环与事件驱动是如何运行的

这一节要把整条链路讲清楚：**消息进来以后，OpenClaw 到底怎么让它有序地跑起来。**

### 2.1 先分清三个词：message、session 和 lane

理解消息系统，最容易混的就是这三个词。先一句话分清：

| 词 | 可以把它理解成什么 |
|----|--------------------|
| `message` | 一条刚刚到达的新消息 |
| `session` | 一段持续合作共用的会话上下文 |
| `lane` | 这段会话在队列里的专属执行通道 |

先看 `message`。它就是一次新输入，可能是你发来一句话，也可能是 heartbeat 触发了一次巡检，还可能是 cron 到点后投递的一次任务。

再看 `session`。它不是“一条消息”，而是这一段合作的长期现场。你们前面聊过什么、刚才看过哪些文件、上一步工具输出是什么，这些通常都挂在 session 上。

最后看 `lane`。`lane` 直译就是“泳道”，你可以把它理解成：**这个 session 在队列系统里自己的一条处理通道。** 消息不是直接拿去执行，而是先根据自己属于哪个 session，被送进对应的 lane。

然后系统遵守一条非常重要的规则：

```text
同一个 lane 里的消息串行执行
不同 lane 之间可以并行执行
```

这条规则几乎是整个消息系统的地基，因为它同时解决了两件事：同一会话里的前后关系不会乱，不同会话又不至于互相堵住。

可以把它画成这样：

```text
lane A（会话 A）:  消息1 → 消息2 → 消息3
lane B（会话 B）:  消息4 → 消息5
lane C（会话 C）:  消息6 → 消息7
```

其中，A 内部严格按顺序跑，B 和 C 可以和 A 同时推进，整个系统还有全局并发上限，避免同时开太多运行把资源打爆。

这里还可以顺手补一个容易忽略的点：lane 不一定只服务“用户聊天”。在 OpenClaw 的实际运行里，定时任务、某些子任务，甚至某些系统级跟进，也可能落到各自的 lane 里。这样做的目的很简单，就是让“不同来源的事情”也能有自己的秩序，而不是全都挤在一条线上。

所以这一层最重要的理解不是“OpenClaw 有队列”，而是：**OpenClaw 用 session 把现场分开，再用 lane 把每个现场的顺序守住。**

### 2.2 同一个 lane 里来了新消息，系统怎么决定怎么接

分了 lane 以后，还剩下一个特别现实的问题：同一个 lane 里，如果 Agent 正在忙，这时又来了一条消息，怎么办？

这时就轮到 queue mode 出场了。所谓 queue mode，你可以先把它理解成：**系统接到“忙中插入的新消息”时，该采取什么态度。**

对初学者来说，先记住四个最常见的模式就够了：

| 模式 | 系统大致会怎么做 | 适合什么情况 |
|------|------------------|--------------|
| `collect` | 先让当前任务跑完，再把等待消息合并处理 | 普通补充、默认情况 |
| `followup` | 明确排到当前任务后面 | “这一件做完，再做下一件” |
| `steer` | 尽快把新消息送进当前运行，引导它改方向 | “先别做那个，先看这个” |
| `interrupt` | 直接打断当前运行，优先处理新消息 | 紧急停止、方向完全错了 |

如果还是抽象，可以用更像人类协作的理解：

- `collect`：你先把手上的动作收完，我把新信息放旁边。
- `followup`：这件事不急，但顺序不能错。
- `steer`：我不是要推翻你，我是要你现在立刻拐个弯。
- `interrupt`：先停，当前这条路已经不该走了。

这里最容易误解的一点是：这些模式不是“谁更高级”，它们只是对应了不同的协作意图。很多时候，默认的 `collect` 反而是最稳的，因为它不会在任务还没收尾时频繁打断现场。只有在确实需要立即纠偏时，`steer` 和 `interrupt` 才更有价值。

官方资料里还能看到更细的变体，比如 backlog 类模式。你先不用一上来就记全。对大多数使用场景来说，先把上面四个弄明白，已经够用了。

### 2.3 高峰期为什么不会彻底挤爆：防抖、容量和溢出策略

光有 queue mode 还不够，因为真实系统里，消息不一定是一条一条慢慢来的。它可能突然一口气冲进来很多条。

比如：

- 用户连续发了五句补充；
- 某个群聊一下子刷了很多消息；
- 某个自动化场景在短时间内触发了多次事件。

如果系统对每条都立刻开一次运行，结果通常不会更快，反而更乱。所以消息系统还有一层更“后台”的兜底策略。

最常见的是这三个概念：

| 机制 | 它在解决什么 |
|------|--------------|
| `debounceMs` | 短时间内连发很多消息时，先等一小会儿再合并处理 |
| `cap` | 给等待队列设上限，避免越堆越长 |
| `drop` | 如果已经超了上限，决定丢旧的、丢新的，还是先压缩再保留 |

先说 `debounceMs`。它就是我们平时说的“防抖”。意思是：如果很短时间内连续来了几条消息，系统先别急着每条都单独开工，而是等一个很小的窗口，看会不会还有补充。

这样做的好处很直接：

- 减少无意义的频繁启动；
- 把几条本来就属于一组的消息合起来理解；
- 降低“刚开始跑就又被新消息改方向”的概率。

再说 `cap`。它本质上是一个上限，因为任何系统都不可能无限堆消息。没有上限，就意味着高峰期可能把内存、上下文和运行时间一起拖垮。

最后是 `drop`。这个词听起来有点可怕，但它其实是在做一个很现实的选择：**如果队列已经满了，系统总得决定到底保留什么。**

常见思路一般有三类：

- 丢最旧的；
- 丢最新的；
- 先把旧消息压缩成摘要，再把重点留下来。

你不需要死记每一种策略的细节，更重要的是理解：OpenClaw 不是天真地假设“消息量永远刚刚好”，它是按真实世界去设计的。真实世界里，洪峰一定会来，所以系统必须有办法在高峰期优雅退化，而不是当场失控。

### 2.4 心跳机制：没人发消息时，Agent 为什么还能自己动

到这里，消息循环已经能解决“有人发消息时怎么处理”。但还有一个更本质的问题：**如果没人发消息，Agent 还能做什么？**

如果答案是“什么都不做”，那它仍然更像一个被动工具。你不叫它，它就一直沉默。OpenClaw 的 heartbeat 机制，就是为了解决这个问题。

所谓 heartbeat，可以直接理解成：**系统按一定节奏把 Agent 叫醒，让它做一次巡检。**

这里最关键的文件是 `HEARTBEAT.md`。它不是代码，更像一张“定期检查清单”。例如：

```markdown
# Heartbeat checklist

- 看一下有没有超时没回的消息
- 检查后台任务有没有卡住
- 如果今天有要跟进的事情，顺手提醒一次
```

这套设计最重要的一点是：系统只负责“定时叫醒”，你负责“醒来以后看什么”。机制和内容是分开的，所以你改 `HEARTBEAT.md`，下次心跳就会按新清单跑。

而 heartbeat 也不是“时间一到就无脑吵醒你”。运行时通常还会结合一些保护条件，比如当前是不是太忙、最近是不是刚处理过消息、是不是处在允许主动打扰的时段。目标只有一个：**主动，但别烦人。**

可以把这些条件理解成“心跳的礼貌”：

- 当前会话正忙时，先别硬插；
- 刚有人来回说了很多句时，先别重复打扰；
- 不在允许主动提醒的时间段时，先安静待着。

这里还有一个非常关键的约定必须记住：`HEARTBEAT_OK`。它的意思可以直接翻译成：**“我检查过了，没什么值得打扰你的。”**

这个约定的价值很大，因为主动系统最怕的就是太吵。如果每次心跳都给你发一句“我看过了，一切正常”，你很快就会把所有通知都当空气。

所以 heartbeat 不是“定时发消息”，它更像：

```text
按时醒来
→ 快速检查
→ 没事就静默结束
→ 有事才真正打扰用户
```

这也是 `HEARTBEAT_OK` 存在的意义：让“安静”成为正式设计，而不是随缘行为。

### 2.5 Heartbeat 和 Cron 很像，但它们解决的不是同一类问题

初学者最容易把 heartbeat 和 cron 混在一起，因为它们看起来都像“定时触发”。但它们的思路其实不一样。

先用一句最短的话分清：

- **heartbeat** 更像巡逻；
- **cron** 更像闹钟。

具体一点看：

| 维度 | Heartbeat | Cron |
|------|-----------|------|
| 关注点 | 持续巡检有没有新情况 | 在某个准确时刻执行某件事 |
| 上下文 | 更贴近主会话，适合延续已有现场 | 更适合独立任务，也常见独立会话 |
| 输出风格 | 倾向于没事就安静 | 倾向于按计划执行并产出结果 |
| 典型问题 | “最近有没有需要我主动提醒的？” | “周一早上九点帮我发周报” |

所以，如果你想要“定期看看有没有异常”，用 heartbeat；如果你想要“某天某时一定执行一次”，用 cron。两者不是替代关系，更像分工关系。甚至很多真实场景里，它们会配合出现：

```text
heartbeat 负责日常巡检
cron 负责准点执行
```

还有一点很重要。cron 常常更适合那些和具体聊天上下文关系没那么紧的任务，而且它可以跑在专门的独立会话里，比如你会在实现里看到类似 `cron:<jobId>` 这样的隔离现场。

这件事的意义很大。因为一旦 cron 跑在独立会话里，它就不会把“定时生成周报”这种后台任务，硬塞进你正在聊天的主现场里。该独立的独立，该连续的连续，这正是消息系统分 lane、分 session 的价值。

而 heartbeat 更适合“需要参考这段合作历史”的事情。比如看看这个会话里是不是有人等太久了，检查之前答应跟进的事情有没有新进展，顺着当前合作上下文做一次轻量巡检。

如果把第二节收成一句话，就是：**OpenClaw 不是把消息一股脑交给模型，而是先按 session 分现场、按 lane 守顺序、按 queue mode 决定插队方式，再用 heartbeat 和 cron 补上时间维度。**

## 第三节 知道机制以后我们应该如何做

前两节讲的是原理。这一节讲更实际的：**如果你真的开始用这套系统，应该怎么配、怎么用，才不容易把自己绕进去。**

### 3.1 默认策略先求稳，不要一上来就追求“最灵活”

很多人第一次碰到这些配置项，直觉都会是：“既然系统这么强，是不是应该把最灵活的模式都打开？”

通常不是。对刚开始使用的人来说，最重要的不是“它能不能立刻聪明地改路”，而是：**它的行为是不是稳定、好猜、可复盘。**

所以更稳的起步方式通常是：

| 项目 | 起步建议 |
|------|----------|
| 默认 queue mode | 先用 `collect` |
| 并发 | 先保守，不要把全局并发开太高 |
| heartbeat | 先写很短的检查清单 |
| cron | 只给真正需要准点执行的任务 |

为什么先用 `collect`？因为它最符合大多数人的直觉：当前任务先做完，新的补充先记着，做完以后再统一接上。这样更稳定，也更容易观察。等你熟悉以后，再根据具体场景引入 `steer` 或 `interrupt`。

### 3.2 四个常用模式，实际应该怎么选

如果你只想记一个最实用的判断法，可以记下面这张表：

| 你的真实意图 | 更适合的模式 |
|--------------|--------------|
| “我只是补充一点信息，你先做完手上的” | `collect` |
| “顺序别错，这件做完再做下一件” | `followup` |
| “别停，但请马上按我说的新方向调整” | `steer` |
| “先停，现在这条路已经不该走了” | `interrupt` |

这里面最容易混的是 `steer` 和 `interrupt`。区别其实很朴素：`steer` 是“边走边拐弯”，`interrupt` 是“先刹车，再重来”。

如果当前运行还有保留价值，用 `steer`。如果当前运行已经明显错了，继续跑只会浪费时间，用 `interrupt`。

而 `followup` 的价值在于“把顺序说死”。它适合那种非常明确的任务链，比如：

```text
先跑测试
→ 测试跑完以后再整理失败用例
→ 整理完以后再写修复建议
```

这种时候，`followup` 往往比 `collect` 更明确，因为它不只是“等会儿再说”，而是“这件事一定排在后一步”。

### 3.3 HEARTBEAT.md 不要写成愿望清单，要写成检查清单

这是初学者特别容易踩的一个坑。很多人第一次写 `HEARTBEAT.md`，会忍不住写成这样：

```markdown
- 变得更主动一点
- 尽量多帮助用户
- 有空就关注各种可能的机会
```

这种写法看起来积极，但不太好用，因为它太虚了。心跳最适合处理的，不是口号，而是**低成本、可执行、能快速判断有没有结果**的检查动作。

所以更稳的写法通常是：

```markdown
- 看看是否有超过 2 小时未回复的会话
- 检查后台进程有没有退出或报错
- 如果今天有到点的提醒，就发出去
```

这些句子有个共同点：能检查，能判断，能执行，没结果时可以安静结束。这正是 heartbeat 最擅长的任务形状。换句话说，**heartbeat 不是拿来证明“我很主动”的，而是把真正值得主动做的那一点点事稳定做起来。**

### 3.4 让 heartbeat 和 cron 分工合作，而不是互相顶替

一个很实用的组合方式是这样的：

```text
heartbeat：
  负责轻量巡检、跟进、提醒

cron：
  负责准点执行、报表、例行任务
```

比如一个学生项目场景：heartbeat 每隔一段时间看一下，有没有实验跑完、有没有同学发来新消息、有没有待跟进事项；cron 每天晚上固定整理一次实验结果摘要。

这时两者就各干各的：heartbeat 保持“活着”，cron 保持“准时”。

反过来，如果你拿 cron 去做持续巡检，通常会很僵，因为 cron 更像一个固定时刻的锤子。它擅长“到点就做”，但不擅长“围着当前会话持续观察有没有新变化”。

同样，如果你拿 heartbeat 去承担所有定时任务，也会变得混乱，因为 heartbeat 的本意是巡检，不是时间表。所以更好的思路不是二选一，而是：**该巡的交给 heartbeat，该准点的交给 cron。**

### 3.5 排障时先看“消息有没有排对队”，再看“模型为什么没做好”

很多人遇到问题时，第一反应是：“是不是模型理解错了？”

有时候确实是，但在消息系统这一层，更常见的原因其实是前面几步就已经乱了。所以更稳的排障顺序通常是：

1. 先看这条消息是不是进了正确的 session；
2. 再看它是不是进了正确的 lane；
3. 再看当前 queue mode 是否符合你的真实意图；
4. 然后看 heartbeat 或 cron 有没有被跳过、延后或静默处理；
5. 最后再去怀疑模型本身的判断。

你可以把它记成一句顺口的话：**先看队排没排对，再看人做没做好。**

比如：

- 你以为系统“没回你”，其实它还在前一个长任务后面排队；
- 你以为 heartbeat “失效了”，其实它检查完以后正常返回了 `HEARTBEAT_OK`；
- 你以为 cron “没执行”，其实它跑在独立会话里，结果不在你当前聊天窗口里。

这些都不是模型理解问题，而是运行机制问题。一旦把这个排障顺序建立起来，你会发现很多问题都能更快定位。

如果把第三节收成一句话，就是：**对初学者来说，最好的用法不是一开始把系统调到最灵活，而是先让它稳定、安静、可预期，再一点点放开更主动的能力。**

## 本章小结

这一章真正讲清楚了两件事。

第一，OpenClaw 为什么不能继续停留在“收到消息就答一句”的阶段。因为只要任务开始持续推进，系统就一定要处理并发、顺序、上下文和时间这几件事。

第二，OpenClaw 是怎么把这些事接住的。它不是靠一个神奇提示词解决问题，它靠的是一整套运行机制：

| 机制 | 它负责什么 |
|------|------------|
| queue | 把新消息先整理，再进入执行 |
| session | 把不同合作现场分开 |
| lane | 守住同一现场里的执行顺序 |
| queue mode | 决定忙的时候该怎么接新消息 |
| heartbeat | 让 Agent 在没人找时也会定期巡检 |
| cron | 让确定性任务在准确时刻执行 |

所以消息循环与事件驱动的意义，不只是“系统更工程化了”。它真正改变的是 Agent 的工作方式：从一次性回复，变成持续协作；从被动等待，变成按节律推进；从谁先到就处理谁，变成先排队、再判断、再执行。

这就是为什么第五章是“心跳”。因为一个真正能协作的 Agent，不只是会说、会做，它还必须会按自己的节律稳定运转。

如果你把这一章真的吃透，后面很多看起来分散的配置项就会突然变得顺眼。你会知道：它们不是零碎开关，而是在共同回答三个问题。

- 这条消息该不该现在处理；
- 这件事该放进哪个现场里处理；
- 这次主动动作到底值不值得打扰用户。

消息循环与事件驱动，本质上就是把这三个问题做成一套长期可运行的系统答案。
