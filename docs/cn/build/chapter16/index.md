# 第16章 多 OpenClaw 协作（以 HiClaw 为例）

本章介绍 HiClaw 如何解决单 Worker 无法应对复杂项目的问题，让读者理解这套方案的价值和运作方式。

---

## 一、先从一个具体困境开始

### 小明的单 Worker 困境

小明是个独立开发者，最近想用 AI 辅助做一个全栈项目。他打开 Claude，开始了一个对话：

**第1天**：小明让 AI 写后端 API，聊得很顺利，完成了用户认证模块。

**第3天**：小明回来继续，发现 AI 好像忘了最初的数据库设计约定，不得不重新解释一遍。

**第5天**：小明想并行推进前端页面，但一个对话只能串行处理，他不得不另开一个新对话，两个 AI 各自为政，代码风格不一致。

**第10天**：小明想检查上周做的功能，翻了500条聊天记录，还是找不到当时确定的接口协议。

**第15天**：项目快收尾了，小明想让 AI 帮忙写测试文档，但 AI 已经记不得最早的需求文档长什么样了。

### 问题在哪

小明的困境不是 AI 不够聪明，而是**单 Worker 的架构局限**：

| 问题 | 表现 |
|-----|------|
| 上下文失忆 | 聊到后期，早期约定被淹没 |
| 无法并行 | 一个对话只能做一件事 |
| 过程黑箱 | 人类难以介入、难以监督 |
| 无法追溯 | 聊天记录翻不到头 |

HiClaw 就是为解决这类问题而设计的。

---

## 二、HiClaw 的解决思路

HiClaw 的思路很简单：**像管理真实团队一样管理多个 AI Worker**。

不是开多个对话各干各的，而是：
- 一个项目经理（Manager）统筹全局
- 多个团队成员（Worker）分工执行
- 在共享空间里记录任务，而不是只存在聊天记录里
- 人类可以随时加入群聊，查看进度、补充要求

下面四个机制支撑起这套协作方式。

---

### 机制1：双轨通信，消息看得见，文件留得住

**没有它时的痛点**

任务全靠聊天交代，三天后问做到哪了，要翻几百条消息。任务需求改了三次，分不清哪个版本是最终版。

**HiClaw 的做法**

把工作分成两条线：

| 线 | 工具 | 用途 | 类比 |
|---|------|------|------|
| 消息线 | Matrix Rooms | 通知、追问、协调 | 项目微信群 |
| 文件线 | MinIO | 存任务需求、存结果、存状态 | 团队共享盘 |

举个例子：
- Manager 在群里说：有个新任务，需求在 `shared/tasks/task-001/spec.md`
- Worker 收到通知，去共享盘读需求文件
- Worker 做完，把结果写到 `shared/tasks/task-001/result.md`
- Worker 回群里说：做完了，结果在文件里

聊天只负责通知，真正的内容落在文件里。这样任务有固定地址，不会淹没在聊天记录里。

在 HiClaw 里，通信线、文件线、能力线是分开的：
- 人类、Manager、Worker 通过 Matrix 房间协作
- Worker 通过共享文件同步读写任务资料
- Worker 通过 Higress 调用 LLM、MCP 等外部能力

这样做的好处是：沟通过程可见，任务资料可追踪，外部能力和密钥还能集中管控。

---

### 机制2：任务对象化，口头指令变成可追踪的卡片

**没有它时的痛点**

Manager 在群里说：做个登录功能。Worker 理解为写个表单，做完才发现要的是 OAuth 登录。需求理解偏差，没有验收标准。

**HiClaw 的做法**

每个任务都是一个对象，有固定格式：

```
shared/tasks/task-001/
├── meta.json     # 任务状态、负责人、优先级
├── spec.md       # 需求文档、验收标准
└── result.md     # 结果产出、完成说明
```

Manager 创建任务时必须写 `spec.md`，把需求写清楚。Worker 做完必须写 `result.md`，说明完成了什么。状态存在 `meta.json` 里，任何人随时可查。

**效果**
- 以前：任务指令散落在100条消息里
- 现在：打开 `spec.md` 就能看到确定的需求，看 `meta.json` 就知道状态

![HiClaw 任务流转图](./images/task-lifecycle.svg)

这张图强调的是：聊天负责沟通，任务目录负责承载任务本体。需求变化时，Manager 更新任务卡片，Worker 按最新文件继续执行。

---

### 机制3：Worker 身份定义，每个 Worker 都有角色档案

**没有它时的痛点**

如果 Worker 都是"通用 AI"，没有角色区分，前端 Worker 和后端 Worker 能力一样，容易互相干扰。搞不清楚哪个 Worker 能访问生产环境，哪个只能访问测试环境。

**HiClaw 的做法**

每个 Worker 创建前都要先定义身份文件（`SOUL.md`），包含：
- 叫什么名字、什么角色（前端/后端/测试）
- 负责什么类型的任务
- 有什么权限限制

Manager 创建 Worker 时读取这个文件，生成对应的配置。Worker 从第一天起就有明确的角色定位。

这类似于给新员工发入职档案：先定义好他是前端工程师还是测试工程师，再给他分配具体的任务。

**效果**
- 以前：所有 Worker 都是"通用 AI"，权限混乱
- 现在：每个 Worker 有明确角色，该干什么、能访问什么，一目了然

---

### 机制4：分层边界，Manager 管调度，Worker 管执行，Higress 管权限

**没有它时的痛点**

如果5个 Worker 都拿着同一个 OpenAI Key，一个泄露全完蛋。某个 Worker 被攻破，攻击者拿着你的 GitHub Token 删库。

**HiClaw 的做法**

把系统分成四层，各司其职：

| 层 | 角色 | 职责 |
|---|------|------|
| 控制层 | Manager | 决定谁做什么任务，但不直接执行 |
| 执行层 | Worker | 只干活，不持有真实密钥 |
| 状态层 | MinIO | 存任务、存结果、存配置 |
| 能力层 | Higress AI 网关 | 统一管理外部 API 调用和密钥，按 Worker 粒度授权 |

这套设计的要点：
- Worker 只带内部凭证（consumer token），不带 OpenAI Key 或 GitHub PAT
- 真实密钥存在 Higress，Worker 通过 Higress 间接调用外部服务
- Manager 可以决定哪个 Worker 能调哪个 API，随时可撤销

**效果**
- 以前：5个 Worker 共用一把钥匙，丢了全完
- 现在：每个 Worker 有自己的门禁卡，Higress 决定他能开哪扇门

![HiClaw 安全边界图](./images/security-boundary.svg)

这张图要表达的重点是：Worker 并不直接持有 GitHub PAT 或 LLM API Key，它只持有内部 token；真正的外部密钥和授权规则集中放在 Higress 一侧。

## 三、整体架构一张图

以前后端开发协作为例：

![HiClaw 协作总览图](./images/hiclaw-overview.svg)

你可以先记住这三条线：
- **通信线**：人类、Manager、Worker 在 Matrix 房间里协作
- **文件线**：任务说明、状态和结果落在共享存储里
- **能力线**：Worker 通过 Higress 访问 LLM、MCP 和其他受控能力

数据流动：
- Manager 通过任务文件（MinIO）控制 Worker 做什么
- Manager、Worker、人类在 Matrix 房间里可见协作
- 任务需求和结果落在共享盘（MinIO），随时可查
- Worker 调用外部能力时，再通过 Higress 做统一代理和权限控制

设计原则：
- 不是"去中心化"，而是"分层中心化"：Manager 统一指挥，Worker 保持独立边界
- 不是"黑箱调度"，而是"透明协作"：所有调度动作在人类可见的房间里发生

## 四、这套方案适合你吗

### 值得考虑的情况

- 项目要持续多天，不是一次性问答
- 需要多个角色并行（前端、后端、测试、文档）
- 人类想随时查看进度、介入修正
- 不想让 AI 直接接触生产环境密钥

### 不适合的情况

- 偶尔问几个问题、写几行代码
- 不想维护 Matrix、MinIO 等额外组件
- 追求极简，一个 CLI 搞定所有
