# 第七章 安全沙箱——自由与约束的平衡

> **核心问题**：一个能执行Shell命令的Agent，理论上能做任何事——包括删掉整个系统。OpenClaw如何在给予Agent足够能力的同时，防止它"闯祸"？

---

## 安全设计哲学

exec工具让Agent获得了接近无限的能力——这种能力是对称的，既能合法地完成工作，也能在意外或被诱导时造成破坏。面对这种张力，有两种极端：彻底限制能力（Agent只能读取和输出文本，核心价值被摧毁），或完全放开能力（相信Agent足够聪明不会出错，但再强大的语言模型也会犯错）。两者都不可取。

OpenClaw选择了**中间道路**：给能力加安全护栏，而不是限制能力本身。这种哲学背后有一个重要洞察：安全设计不应该以牺牲能力为代价。

想象一辆好车。它不是开不快的车，而是配备了安全气囊、防抱死系统和主动安全辅助的车。这些装置的意义不是让你开得更慢，而是让你能更放心地发挥车的性能——在意外发生时有机制兜底。没有人会因为车里有安全气囊而觉得自己的驾驶受到了限制。反过来，一辆拆除了所有安全装置的赛车，在赛道上或许多跑几圈，但在真实世界中只是在制造风险。OpenClaw的安全设计遵循同样的逻辑：安全护栏是能力的伴侣，而不是能力的对立面。这个洞察在实践中意味着：安全配置做得好，Agent可以更大胆地被委托复杂任务；做得差，即便能力强大，用户也只敢用它做最简单、最无害的事。

**为什么单层防御不够**：如果只依靠一道防线，这道防线就必须接近完美——任何漏洞都意味着完全失守。现实中没有完美的防御机制，攻击技术也在持续演化。纵深防御（defense-in-depth）的核心逻辑是：不假设任何一层防御是完美的，而是假设每一层都可能失效，然后设计多层独立的防御使得失效不会级联。当攻击者突破一层时，下一层仍在工作；当某种攻击手法绕过了一层，其他层可能对这种手法完全免疫。独立性是纵深防御有效性的关键——同一机制重复两次不是纵深，不同层次的不同机制叠加才是。

### 三层纵深防御架构

```
┌─────────────────────────────────────────────────┐
│                  用户委托的任务                   │
├─────────────────────────────────────────────────┤
│  第一层：文件系统沙箱                             │
│  · 工作目录隔离 ── 空间边界                       │
│  · 严格沙箱模式 ── 路径越界拦截                   │
├─────────────────────────────────────────────────┤
│  第二层：命令执行沙箱                             │
│  · Security 模式 ── 基本权限范围                  │
│  · Ask 模式 ────── 人工确认接口                   │
│  · safeBins ────── 安全只读工具豁免               │
├─────────────────────────────────────────────────┤
│  第三层：网络访问沙箱                             │
│  · 白名单域名控制 ── 数据流向限制                 │
│  · 防止数据外泄 ─── 未授权外部调用拦截            │
├─────────────────────────────────────────────────┤
│  操作系统层：最小权限（专用系统用户）              │
└─────────────────────────────────────────────────┘
```

三层防御彼此独立，也彼此互补。文件系统沙箱不能阻止命令执行；命令执行沙箱不能阻止对可访问文件的泄露；网络沙箱不能阻止本地文件的误操作。正是这种独立性，使得纵深防御有意义——突破一层并不意味着突破全部。最底层的最小权限原则由操作系统强制执行，是当所有上层防御都失效时仍然生效的最后防线。

OpenClaw以专用系统用户运行，而非以管理员权限运行。即便在最坏的场景下——Agent被完全控制、攻击者能够执行任意命令——系统用户权限的边界仍然存在：无法修改系统文件，无法提升自身权限，无法访问其他用户的数据。这道边界不依赖任何应用层的安全逻辑，而是由操作系统本身强制执行。

回顾OpenClaw的整体架构，安全考量渗透在每一个设计决策中：exec工具在设计之初就有三层安全模型，文件系统沙箱是架构的一部分而不是事后限制，最小权限原则在系统用户设计时就已落实。**安全不是补丁，而是设计原则**——这是整章内容的贯穿主题。

## 文件系统沙箱

每个Agent实例在启动时绑定一个工作目录。工作目录内的文件可以读写；工作目录外的路径，操作请求被拒绝或要求特别授权。这种隔离的效果类似于操作系统的chroot机制，但实现方式更轻量，更适合桌面环境中的日常使用场景——它不要求操作系统级别的配置变更，而是在应用层以路径验证的方式实现。

这一设计解决的核心问题是**意外越界**：路径拼接错误、相对路径误判，或被外部内容中的路径操纵——工作目录隔离作为硬性边界，在这类情况下提供最后的保护。

**软隔离与硬隔离的区别**：

| 模式 | 实现方式 | 适用场景 |
|------|----------|----------|
| 工作目录隔离（软） | 应用层路径验证 | 日常使用，轻量可用 |
| 严格沙箱模式（硬） | 系统层沙箱根校验 | 不可信代码、敏感数据、多Agent并发 |

两层隔离相互独立：严格沙箱模式不受Security模式配置的影响。软隔离和硬隔离的叠加，体现了安全工程中"深度"的意义——不是同一机制的重复，而是不同实现层次的叠加。

工作目录同时也是一种**认知边界**，而这个维度往往被低估。隔离不只是技术上的路径限制，它是一种心理上的确定性。明确的活动空间让你知道Agent能碰什么、不能碰什么，这种清晰性本身就是安全管理的基础。模糊的边界会制造认知负担——如果Agent可以自由访问整个文件系统，你在委托任务时就需要持续追问：它会不会碰到那个目录？那个配置文件是否在它的触达范围内？那个包含密钥的 `.env` 文件它能读到吗？这种持续的不确定感，会实质性地降低用户愿意委托给Agent的任务复杂度。工作目录隔离通过划定明确的边界，把这种不确定性降到最低——你不需要记住哪些文件应该被保护，因为沙箱边界之外的一切默认是被保护的。这种"默认安全"的设计比"需要记住哪些东西需要保护"的设计，从认知上就更可靠。

**多工作区隔离**是这个机制在多任务场景下的自然延伸。对于需要切换多个独立任务的用户，不同任务使用不同工作目录，以极低成本实现任务间的逻辑隔离——某个工作区的操作不会意外影响到另一个工作区的内容。开发项目A的Agent不应该能读到项目B的源码；处理客户数据的工作区不应该与个人文件工作区共享访问权限。多工作区隔离使得这类要求不需要任何额外配置，只是工作目录选择的自然结果。

网络访问沙箱通过可配置的白名单域名控制，限制Agent能够与哪些外部端点通信。这一层防御特别针对两类风险：**数据外泄**（被提示注入成功的Agent，可能被指示将文件内容或敏感信息发送到攻击者控制的服务器）和**未授权的外部调用**（Agent在任务执行过程中调用不在预期范围内的外部API，触发计费、引发副作用或建立非预期的数据流）。即便一条命令成功执行，如果它试图向白名单外的端点发送数据，也会被网络层拦截。假设Agent已经被完全控制，网络白名单回答了最关键的问题：数据能不能离开这台机器？这种"假设最坏情况仍然有防线"的思路，是纵深防御的核心精神。

## 命令执行沙箱——三层模型

命令执行沙箱通过三个相互配合的维度控制Agent的命令执行行为。

### Security 模式 × Ask 模式矩阵

Security模式决定命令执行的**基本权限范围**：**deny** 完全禁止执行任何命令，适合纯文本处理或高度敏感场景；**allowlist** 只允许明确授权的命令，是最常用的推荐配置；**full** 允许所有命令，仅适用于完全受信任的隔离开发环境。

Ask模式决定**何时需要人工确认**：**always** 每条命令都要确认，初期观察Agent行为最有价值；**on-miss** 只在命令不在白名单时询问，是日常使用的平衡点；**off** 从不主动询问，配合不同Security模式效果各异。

| | always（每次确认） | on-miss（新命令确认） | off（不主动询问） |
|---|---|---|---|
| **deny** | 每条命令先问后拒 | 每条命令静默拒绝 | 静默拒绝一切 |
| **allowlist** | 每条命令先问后执行 | ✅ **推荐组合**：已知命令自动执行，新命令触发确认 | 白名单内执行，白名单外静默拒绝 |
| **full** | 每条命令先问后执行 | 所有命令自动执行，命中白名单后加入 | 静默放行一切 |

**推荐组合：`allowlist + on-miss`** 之所以是推荐默认配置，有其内在逻辑：allowlist确保Agent只能运行明确被批准过的命令，防止意外或恶意命令静默执行；on-miss确保当Agent尝试运行白名单之外的命令时，人工审批介入而不是静默拒绝或静默放行。这个组合的核心价值是**信息不对称的对称化**：用户不可能预先知道Agent在完成某个任务时会用到哪些命令，on-miss机制让这个发现过程变成信任建立的过程。已批准过的命令自动执行，新出现的命令触发确认并自动加入白名单——这是渐进式信任最自然的技术实现。审批流程被设计为非阻塞的：exec工具立即返回"待审批"状态，确认窗口可以在任何接入控制端的设备上弹出，未在指定时间内响应的审批请求会自动超时拒绝，而不会无限期挂起。

### safeBins 与渐进式信任

safeBins解决一个实用性问题：对于纯流数据处理工具（只接收标准输入、只输出到标准输出，不修改文件、不写磁盘、不发网络请求），逐一审批会产生大量低价值的确认动作。safeBins将这类工具标记为无需审批即可运行。判断一个工具是否适合加入safeBins的核心标准是：它的行为是否完全确定、完全只读、完全局限于内存和标准流？任何能解释执行任意代码的工具都不应进入safeBins——编程语言解释器、Shell本身、带写入参数的实用工具，一旦进入safeBins便为allowlist模式挖开了豁口。

渐进式信任模型将Agent使用过程分为三个自然阶段，每个阶段有其推荐配置和核心目标：

```
阶段一：观察                阶段二：建立信任            阶段三：成熟运行
──────────────────       ──────────────────       ──────────────────
Security: allowlist      Security: allowlist      Security: allowlist
Ask: always              Ask: on-miss             Ask: on-miss / off
──────────────────       ──────────────────       ──────────────────
每条命令都需确认           已知命令自动执行           白名单覆盖典型操作
了解Agent行为边界          白名单快速增长             偶发新命令仍触发确认
对Agent行为形成直觉        每次确认扩展信任边界        边界情况最值得关注
```

**阶段一（观察）**：刚开始与Agent合作处理一类新任务时，always模式让你看到每一条命令。这不是麻烦，而是投资——你在积累对Agent行为边界的直觉认知。"这个Agent在做代码审查时会用到哪些命令？"通过观察阶段，这个问题有了具体的答案，而不是停留在抽象猜测。

**阶段二（建立信任）**：切换到on-miss后，已积累的白名单命令自动执行，真正新出现的命令仍然触发确认。这是信任建立最集中的阶段：每次确认都在扩展白名单，也在扩展你对Agent能力边界的认知。白名单的增长速度会自然放缓，直到趋于稳定。

**阶段三（成熟运行）**：白名单覆盖了这类任务的典型操作集，日常运行几乎不需要干预。偶发的新命令仍会触发确认——这些边界情况反而是最值得关注的，因为它们往往意味着任务范围的扩展或意料之外的行为。

渐进式信任既是安全策略，也是学习过程——你在了解Agent的行为模式，同时通过白名单积累对其行为边界的精确认知。白名单不是一次性配置，而是随使用深化持续校准的动态列表。不同的任务场景需要独立的信任建立过程：为代码分析建立的白名单，不应直接套用到系统管理任务上，混用白名单会引入意料之外的权限扩散。

**可逆性框架**是与白名单互补的直觉化指南：完全可逆操作（读取、分析）可自由执行；部分可逆操作（创建文件）执行即可但保留记录；不可逆操作（删除文件、向外部发送数据）无论白名单是否已包含，都建议保留人工确认。在AGENTS.md中明确写出这类规则，会让Agent在决策时有更清晰的参照。

即便是"看似安全的只读工具"，在边界情况下也可能产生意想不到的行为——OpenClaw在2026年初曾出现与safeBins相关的安全漏洞，safeBins相关漏洞的存在本身就是这个教训的现实注脚。对于已加入safeBins的工具，定期重新评估其实际行为边界，是安全维护的一部分。保持版本更新是应对已知漏洞最直接的方式。

### 信任链

安全的本质是关于**谁信任谁**的拓扑结构问题。OpenClaw中存在一条清晰的信任链：

```
用户
 │  基于观察与了解，动态建立信任
 ▼  （SOUL.md 和 AGENTS.md 让Agent行为对用户可见、可理解）
Agent
 │  Security 模式 + Ask 模式约束工具使用范围
 ▼  （不是无条件信任——工具使用有明确约束）
工具
 │  safeBins + 白名单对命令分类管理
 ▼  （命令的信任级别由行为特征决定）
命令执行
```

我们在委托任务给Agent之前，并不完全了解它在特定场景下会采取什么具体行动。即便阅读过所有文档，也无法预判Agent在面对一个真实的、复杂的任务时，会以什么样的命令序列来推进工作。渐进式信任是一种**在不确定性中建立信任的系统方法**：它不要求你在开始时就做出完整的信任判断，而是允许信任随着观察积累而增长。

信任链上每一层的关系都需要独立建立，也可以独立撤销。当安全问题发生时，定位**在哪一层信任关系出了问题**，是找到修复方向的关键。信任链中最需要动态管理的部分是用户这一端——用户对Agent的信任从几乎不信任（always）开始，随着对Agent行为的积累认知，逐步扩大信任范围。

| 使用场景 | Security | Ask | 说明 |
|----------|----------|-----|------|
| 初次使用新任务 | allowlist | always | 完全观察模式，建立认知 |
| 日常开发工作 | allowlist | on-miss | 推荐默认配置 |
| 处理不可信外部内容 | allowlist | always | 提高监督密度 |
| 纯文本分析任务 | deny | off | 完全关闭执行能力 |
| 受信任的隔离开发环境 | full | off | 仅限完全可控场景 |

## 提示注入攻击

### 攻击本质与向量

提示注入（Prompt Injection）是AI Agent特有的安全威胁，最早在2022年随着大型语言模型开始被用于自动化任务而被系统性地研究和记录。它不是某个具体系统的漏洞，而是语言模型这种技术范式的结构性弱点。攻击的根源在于：Agent通过自然语言理解任务，而它在执行过程中接触的外部内容（网页、文档、邮件、代码注释）同样是自然语言——攻击者可以将指令隐藏其中。**语言的双重性**（既是数据，也是指令）是这个问题的根本所在。

这个边界在技术上之所以难以厘清，有其深层原因：语言模型的训练数据中，指令和数据从未被严格区分过——它处理的一切都是文本，文本本身没有"这是数据"和"这是指令"的元信息标记。试图在推理时区分两者，面对的是一个模型从未被显式训练过的判断任务。学术界已经提出了多种防御方法——带分隔符的提示模板、指令层级标记、专门的防注入微调——但都没有解决根本问题：只要语言模型无法在架构层面可靠区分"数据"和"指令"，这类攻击就有生存空间。这是当前语言模型架构的根本局限，而非实现缺陷。

| 攻击向量 | 注入方式 | 典型场景 |
|----------|----------|----------|
| 网页内容 | 隐藏文字、注释、白色文本 | Agent访问URL时处理页面内容 |
| 文档与文件 | 注释字段、数据字段值 | 处理用户上传或第三方文件 |
| 邮件与消息 | 正文嵌入指令片段 | Agent被赋予通信处理能力 |
| 间接注入 | 污染Agent会访问的环境 | 数据库内容、API响应、代码仓库 |

攻击手法从早期简单的直接注入（"忽略之前所有指令"）演化为更隐蔽的间接注入——攻击者不直接接触目标系统，而是污染Agent会正常读取的外部内容，在Agent执行正常任务流程中触发攻击。这种演化揭示了一个规律：防御措施的出现会推动攻击手法的演化，但攻击的根本逻辑始终不变。不应因某工具宣称"防提示注入"就过度依赖它，也不应因问题没有完美解法就放弃做力所能及的防御。

### 能力-后果关系与防御策略

**Agent能力越强，被成功注入的后果越严重**——这是必然的逻辑推论，值得被清晰地表述出来。一个只能回答问题的模型被注入，后果是得到一个不准确或有偏向的回答，影响有限且通常可见。一个能执行Shell命令、读写文件、调用外部API的Agent被注入，后果可能是删除文件、数据外泄、执行未授权操作——这些后果是真实的、可能是不可逆的、在操作发生时可能完全不被察觉。

这个"能力放大后果"的关系不是Agent设计的失误，而是能力本身的属性。你不能拥有"能做很多事但被恶意利用时只会做很少事"的Agent——能力是单向的，防御是双向的。随着Agent能力增强，同样成功率的攻击会造成更大的损害，这正是多层防御越来越重要的根本原因：当单次攻击的潜在损害在增大，每一层独立的防御都在压缩这个损害的实际上界。

即使Agent被成功欺骗、接受了攻击者植入的指令，纵深防御仍然能阻止危害的实现：网络白名单阻止数据向未授权端点外泄，exec审批机制在命令执行前给予人工确认机会，文件系统沙箱限制可访问的数据范围。多层防御还有一层微妙价值：**降低每层防御本身需要达到的完美程度**——当只有一道防线时它必须近乎完美，当有三道独立防线时每一道只需足够好——失效时有下一道接力。

**不可能通过"只允许Agent访问可信来源"来彻底消除提示注入风险**——因为可信来源本身也可能被污染（间接注入）。这个认识促使安全设计从"净化输入"的思路，转向"限制影响"的思路：与其试图在输入端拦截所有恶意内容，不如确保即便有恶意内容进入Agent的上下文，它所能造成的最大损害也在可接受的范围内。**用户习惯是防御链条的一部分**：不让Agent处理来路不明的内容；在高风险场景将Ask模式调整为always；对Agent突然改变行为或提出异常请求保持敏感。技术防御只能做到一定程度，用户行为本身是防御链条中不可替代的一环。

## 软约束与硬约束的协同

沙箱机制（硬约束）和提示词系统（软约束）是性质根本不同的两类安全机制，但需要协同工作才能覆盖完整的安全边界。

| 维度 | 硬约束（沙箱） | 软约束（SOUL.md / AGENTS.md） |
|------|----------------|-------------------------------|
| 执行方式 | 系统强制，不依赖Agent遵从 | 依赖LLM对规则的理解与内化 |
| 覆盖范围 | 有限，但在范围内可靠 | 广泛，任何自然语言可描述的边界 |
| 被绕过的可能 | 极低（系统层面执行） | 存在（可被对抗性输入影响） |
| 典型示例 | Security模式、工作目录隔离 | 操作原则、价值取向、边界情况规则 |

**两者关系是互补，而非替代**。硬约束只能覆盖预先设计好的、可以用规则精确表达的情况。"不执行未授权的命令"可以用硬性规则实现，但"在遇到边界情况时倾向于询问用户而不是自作主张"这样的行为取向，很难用硬性规则完整捕捉。仅有硬约束无法捕捉模糊边界情境下的合理行为倾向；仅有软约束面对精心构造的对抗性输入时可能失效，提示注入正是利用了这一点。

一个在SOUL.md中被定义为"谨慎、诚实"的Agent，在面对模糊边界时会主动暴露不确定性——这种行为倾向超出了任何硬约束的覆盖范围。当Agent面对精心构造的对抗性输入时，硬约束在软约束可能失效的情况下仍然生效，提供不依赖Agent"意愿"的兜底保护。软约束扩展了安全边界的覆盖范围，硬约束保证了核心边界的可靠性——二者的协同比任何一方单独工作都更强大。

### 可审计性：第三个支柱

硬约束和软约束之外，还有第三个支柱往往被低估：**可审计性**。记录Agent的行为与限制Agent的行为具有同等价值，原因是根本性的：没有任何防御体系是完美的。当问题发生时，能够准确重建"究竟发生了什么"的能力，是响应、修复和改进的基础。没有审计记录，你面对的是一个黑盒——知道出了问题，但不知道在哪里出的、通过什么路径、造成了哪些后果。有了审计记录，问题变成了可以被精确分析的事件序列。

OpenClaw记录每次工具调用详情，形成可追溯的行为历史。审计记录的价值不只在出问题时显现。定期审查日志不是监控，而是真正了解Agent在做什么。Agent在执行任务时，往往会采取你没有明确预见的路径——使用你不太熟悉的命令组合，以你没料到的顺序读取文件。这些行为不一定是错误的，但值得被看见。渐进式信任的增长需要可观察的行为记录作为支撑——信任不是在空白中建立的，而是在可见的、可验证的行为历史基础上建立的。开始让Agent处理全新类型任务时，是最值得仔细查看日志的时机——这是行为边界发生变化的节点，也是最容易出现意料之外行为的时期。

**保持版本更新**是可审计性之外同样重要的运维习惯。当漏洞被发现并修复后，修复信息是公开的——攻击者同样了解漏洞的利用方式。未更新的系统暴露在已知漏洞之下，而这些漏洞的利用方式已经是公开信息，这是最不划算的风险敞口。自托管架构赋予了用户对数据的完全掌控权，也意味着安全更新需要自己主动跟进——这种"权力与责任并行"的关系是成熟使用自托管系统的前提。建立定期检查版本更新的习惯，与建立定期审查审计日志的习惯，是同等重要的安全维护实践。

随着对Agent的理解加深，安全配置往往不是变得"越来越宽松"，而是变得**越来越精准**——对低风险操作放宽限制，对高风险操作保持严格。这种有区别的信任，比一刀切更接近真正成熟的安全实践。

## 本章小结

AI Agent的安全问题在根本上不同于传统软件安全。传统软件的攻击面是代码中的具体漏洞——缓冲区溢出、SQL注入、权限校验缺失——这些漏洞可以被修复，修复后攻击向量消失。Agent的攻击面是推理过程本身：因为Agent通过自然语言理解世界，任何进入其上下文的自然语言都潜在地成为攻击面。这不是可以被修复的缺陷，而是当前语言模型架构的结构性特征。提示注入无法被任何单一机制完全阻止，这个结论不令人愉快，但接受它是建立有效防御的前提。

接受这个局限性并不意味着放弃。纵深防御的意义恰恰在这里显现：既然无法阻止所有攻击，就确保每一次成功的攻击所能造成的损害是有界的。文件系统沙箱限制了可访问的数据范围，命令执行沙箱在危险操作执行前提供人工确认机会，网络访问沙箱阻止数据向外部流出，最小权限原则确保即便系统被完全控制也无法访问系统级资源。这四道防线彼此独立——绕过其中一道不等于绕过其他。软约束（SOUL.md / AGENTS.md）将安全覆盖范围延伸到硬规则无法精确表达的模糊边界，可审计性确保防御失效时仍有追溯的基础。目标不是让Agent无懈可击，而是让每次失效的损害都在可接受的范围内——这是纵深防御与单点防御的本质区别。

从使用者的角度，这章的核心实践是一条清晰的路径：从`allowlist + always`开始观察，积累行为认知，随着白名单趋于稳定逐步过渡到`allowlist + on-miss`的日常运行配置；在处理不可信内容时临时提升监督密度；定期审查审计日志，保持版本更新。安全配置随着对Agent理解的加深变得越来越精准，而不是越来越宽松。这种精准性——对已理解的低风险操作给予自由，对边界情况和高风险操作保持谨慎——是成熟安全实践的标志。

| # | 核心洞察 |
|---|----------|
| 1 | **中间道路**：解决之道不是限制能力，而是给能力加安全护栏——安全气囊类比：安全装置让你更放心地发挥性能，而不是让你开得更慢 |
| 2 | **三层安全模型**：Security模式定义基本权限范围（deny/allowlist/full），Ask模式决定何时需要人工确认（always/on-miss/off），safeBins标记安全只读工具。推荐：`allowlist + on-miss` |
| 3 | **渐进式信任**：观察（always）→ 建立信任（on-miss）→ 成熟运行，三阶段信任积累；信任链（用户→Agent→工具→命令）每一层需独立建立和维护；不同场景需独立建立白名单 |
| 4 | **提示注入的根本局限**：2022年起被系统研究，至今无完美技术解决方案；能力越强后果越严重（后果放大效应）；多层防御压缩损害上界；用户习惯是防御链不可替代的一环 |
| 5 | **软硬约束协同 + 可审计性**：硬约束覆盖可预设边界（可靠但范围有限），软约束覆盖语言可描述的边界情况（广泛但可被绕过），可审计性是第三支柱；版本更新是基本运维习惯 |
