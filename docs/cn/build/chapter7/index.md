# 第七章 安全沙箱——护栏不是枷锁

> **核心问题**：OpenClaw 怎么在“能做很多事”和“不会把系统搞坏”之间找到平衡？

---

前面几章一路讲下来，我们已经看到：OpenClaw 不只是会聊天，它还会推进任务、调用工具、长期接住来自不同地方的事情。问题也随之变得更硬了：**它既然真的能动手，会不会把事情做过头？**

这个担心一点也不多余。只要 Agent 拿到了 `exec` 这类工具，它能碰到的就不再只是聊天记录，而是文件、进程、网络、外部服务，甚至整台机器的运行环境。能力一旦变强，风险就一定跟着变大。

这时系统很容易走向两个极端：全关掉，安全但没什么用；全放开，强大但很难放心托付。所以这一章要讲的不是“更保守”还是“更激进”，而是：**怎么给能力加上边界，让它既能做事，又不至于乱来。**

这就是“安全沙箱”的位置。你可以先把“沙箱”理解成一个很朴素的概念：**让系统在一个被圈住的范围里行动。** 它不是把能力拿走，而是先把活动范围、可见内容、执行条件和升级通道讲清楚。

---

## 第一节 为什么 OpenClaw 需要安全沙箱

这一节只回答一个问题：**为什么“能动手”这件事，必须从第一天开始就跟“有边界”绑在一起。**

### 1.1 只给能力不给边界，会出什么问题

先看最直接的事实。当 Agent 只能读文字、写回复时，它最坏也不过是答错；但当 Agent 能运行命令时，后果就完全不是一个量级了。它可能误删文件、覆盖配置、启动不该启动的进程，也可能在错误目录里跑破坏性命令，甚至把本地数据带出外网。

可以看一个最简单的对比：

| 能力层级 | 出错后的典型后果 |
|----------|------------------|
| 只聊天 | 回答错、理解错、风格不对 |
| 能读写文件 | 内容被改坏、文件被覆盖 |
| 能执行命令 | 进程、目录、依赖、网络都可能被影响 |

这就是为什么安全问题在 Agent 这里不是“附加项”，而是基础设施的一部分。很多同学第一次听到这里，会下意识地说：“那是不是只要模型足够聪明，就能自己判断哪些命令危险？” 这句话只对一半。模型当然可以帮助判断，但你不能把安全完全押在“它这次会不会判断对”上。

原因很简单：安全不是一次聪明判断的问题，安全是**系统边界**的问题。成熟系统不会只说“请你小心一点”，它会先把路栏上。所以第一个结论是：**只给能力，不给边界，不是在做强系统，而是在做碰运气的系统。**

### 1.2 为什么“彻底禁用执行”也不是答案

但另一边也不能走太远。如果一看到风险，就把所有执行能力都关掉，结果通常是：这个 Agent 变得非常安全，同时也变得非常没用。因为很多真正有价值的任务，本来就离不开“动手”。

比如读项目文件并修改、跑测试、看日志、执行格式化或构建命令、调用本地脚本、与受控的外部服务交互，这些事情本来就不是“嘴上说说”能解决的。如果这些全被关掉，Agent 剩下的就主要还是“告诉你怎么做”，那它和一个只会建议你的聊天模型，又拉不开多大差距了。

这也是为什么“彻底禁用执行”看上去很稳，实际上却不是 OpenClaw 想要的方向。OpenClaw 的目标不是做一个最保守的聊天机器人，而是做一个**真的能帮你推进任务**的 Agent。而“推进任务”这四个字，本来就意味着它要碰文件、碰命令、碰运行环境。

所以问题不是“要不要给它能力”，而是：**给它多少能力，以及这份能力要被什么规则约束。**

### 1.3 安全沙箱真正要解决的核心矛盾

走到这里，这一章的核心矛盾就很清楚了：

| 路线 | 好处 | 问题 |
|------|------|------|
| 全放开 | 能力最强，事情最好做 | 风险太高，几乎无法放心托付 |
| 全关掉 | 风险最低 | 真正能做的事太少 |
| 有边界地放开 | 既能做事，又能被约束 | 设计更复杂，但这是唯一可用的路 |

所以“安全沙箱”并不是一个和能力对着干的东西。它的作用不是说“你别做”，而是说：“你能做，但要在我规定的范围里做；超范围时，要么停下来，要么先经过确认。” 这也是为什么这一章的标题是“护栏不是枷锁”。

再往前一步想，为什么成熟系统通常不只放一道门？因为单层防御太脆了。你不能把全部希望都压在某一次模型判断、某一个审批弹窗，或者某一条 allowlist 规则上。更稳的思路是：**不假设任何一层会永远完美，而是假设每一层都可能出错，于是用多层边界彼此补位。**

如果把第一节收成一句话，就是：**OpenClaw 不是在“安全”和“能力”之间二选一，而是在设计一套机制，让能力可以被可控地放出来。**

## 第二节 安全沙箱是怎样跑起来的

这一节要把机制讲清楚。最容易误解的一点是：很多人会把“安全沙箱”理解成一个单独开关，好像只要开了就安全，关了就不安全。真实情况不是这样。

OpenClaw 的安全更像是几层门：模型先能不能看见这个工具；就算看见了，这个工具能不能在当前策略下被调用；真到执行时，这条命令能不能通过审批和规则；就算执行了，它又能在什么文件和网络边界里运行；遇到极少数必须升级权限的场景，升级通道又怎么被控制。也就是说，安全不是一个点，它是一条链。

### 2.1 第一层：工具先要“看得见”，才谈得上“用不用得上”

先讲第一层，这是最容易被忽略、但其实特别重要的一层：**如果模型根本看不见某个工具，它就不可能去调用它。**

这里先解释两个词。“工具可见性”指的是这次运行里，模型到底能看到哪些工具；“工具策略”指的是系统根据配置，决定哪些工具要被放进当前这次运行。

OpenClaw 这一层常见的配置有三个：

| 配置项 | 可以先把它理解成什么 |
|--------|----------------------|
| `tools.profile` | 一份默认工具包 |
| `tools.allow` | 额外明确放行的工具或工具组 |
| `tools.deny` | 明确禁掉的工具或工具组 |

先看 `tools.profile`。它很像“基础套餐”，你给某类 Agent 一个 profile，相当于先给它一组默认能力。再看 `tools.allow`，它的意思是：如果默认套餐里没有，但我又确实想给它，就在这里补进去。最后是 `tools.deny`，它的意思是：不管默认套餐里有没有，只要我明确不想让它碰，这里一禁掉，它就别想再进来。

这里有一个规则要记住：**`deny` 的优先级通常更高。** 也就是说，先有、后禁，是允许的；一旦禁了，就别指望它还能通过 profile 混回来。可以把这层机制想成“发工具卡”：

```text
先按 profile 发一套基础卡
→ 再用 allow 补几张
→ 最后用 deny 把不该给的抽掉
→ 剩下的，才是真正出现在模型眼前的工具
```

这层设计特别关键，因为它解决的是“收口”。Agent 不是工具越多越好，工具越多越容易选错，风险工具越多边界越难守。所以安全的第一步往往不是“执行时再拦”，而是：**一开始就别把不该给的工具摆到模型面前。**

这里还经常会遇到“工具组”这个说法。你可以先把它理解成“打包好的工具分类”。这样做的好处是，你不用一个个名字去写，而是按类别收口。

这一层还有一个很重要的效果：**减少误用。** 很多安全问题并不是模型“故意乱来”，而是它面前工具太多、太杂、太重叠，最后在几个看上去都能用的选项里选错了一个。

### 2.2 第二层：一条 `exec` 命令到底要过哪些关

到了第二层，才真正进入大家最关心的地方：`exec`。先把这个词讲成人话，`exec` 的意思很简单，就是**让 Agent 真去执行一条命令**。例如：

```text
ls
npm test
git status
python script.py
```

也正因为它真的会执行，所以它必须比普通工具多过几道关。可以先把这条链记成：

```text
模型想调用 exec
→ 系统先看工具策略允不允许
→ 再看当前安全模式允不允许
→ 再看这条命令是否需要审批
→ 真通过后才执行
```

这里最常见的几个词，需要先讲清楚。

**`security`**

它可以先理解成“权限范围有多大”。常见思路通常有三档：

| 模式 | 可以先理解成什么 |
|------|------------------|
| `deny` | 基本不让执行 |
| `allowlist` | 只允许已知、已信任的一部分 |
| `full` | 基本完全放开 |

这三档里，最适合大多数日常协作的，通常不是最左，也不是最右，而是中间的 `allowlist`。原因很简单，它既没有把能力彻底掐死，也没有把边界一口气拆掉。

再看 **`ask`**。这个词可以直接理解成“要不要先问你一声”，也就是审批时机。常见思路一般有：

| 模式 | 它更像什么 |
|------|------------|
| `always` | 每次都先问你 |
| `on-miss` | 已知允许的不用问，碰到新情况再问 |
| `off` | 不主动询问 |

这时你就会发现，`security` 和 `ask` 解决的不是同一个问题：`security` 管的是“能不能”，`ask` 管的是“要不要先确认”。这两个维度叠在一起，才构成了真正的执行策略。

可以看一张很实用的速查表：

| `security` | `ask` | 大致效果 |
|------------|-------|----------|
| `deny` | 任意 | 基本不执行 |
| `allowlist` | `always` | 每次都看得很严，适合观察期 |
| `allowlist` | `on-miss` | 常用组合，已知命令放行，新命令先确认 |
| `full` | `off` | 几乎完全放开，只适合高度受控环境 |

这里面最值得专门讲一下的是 `allowlist + on-miss`。为什么它经常被当成比较平衡的组合？因为它很像一种“渐进式信任”。你已经认可过、也反复见过的命令，不需要次次都问；但新命令冒出来时，系统仍然会停一下，让你决定要不要放过去。这比“次次都问”省心，也比“什么都不问”稳得多。

再往下还有两个容易搞混的词：**审批** 和 **allowlist**。它们不一样。审批说的是这一次要不要先问你，allowlist 说的是这类命令是否已经进入“默认可信集合”。也就是说，审批更像一次临场确认，allowlist 更像一份长期积累的信任名单。

如果你想把这条链看得更具体一点，可以想象这样一个场景：

```text
用户：把测试跑一下
→ 模型决定调用 exec("npm test")
→ 系统先看 exec 这个工具在不在当前工具集合里
→ 再看当前 security 是否允许这类命令
→ 如果 npm 相关命令不在 allowlist 里，就触发 ask
→ 用户确认后，命令才真正执行
```

这个例子能帮你看清一件事：一条命令能不能跑，不是由某一个地方单独拍板的。它通常要连续经过“看不看得见”“策略允不允许”“要不要先确认”这几步。

这也是为什么排障时不能只盯着命令本身看。很多时候命令没有错，错的是它在前面的某一层就已经被拦住了。

### 2.3 `safeBins`、文件边界和网络边界，分别在挡什么

讲到这里，很多人会觉得：“那是不是所有命令都要同样严地过一遍？” 也不是，因为命令的风险并不一样。

这里先讲一个很实用的词：`safeBins`。你可以先把它理解成：**一小组被系统认为相对安全、通常偏只读的小工具。** 比如某些查看、过滤、统计类命令。它们的共同点通常是更像“看”和“整理”，不太像“改”和“破坏”，行为比较确定，而且很多时候只在标准输入输出里流转。

所以 `safeBins` 的本质不是“它们绝对安全”，而是：**它们通常足够简单、足够只读，因此可以走更轻一点的信任路径。** 但这里一定要注意一个边界：不是所有看起来常见的命令都该进 `safeBins`。尤其是那种能解释执行任意代码、能写文件、能联网拉东西、能改变系统状态的命令，都不应该因为“常见”就被当成安全捷径。

换句话说，`safeBins` 解决的是“别为低风险只读动作反复打断用户”，而不是“给一堆命令开后门”。这个边界一定要守住。

接着讲文件边界。很多人一提到沙箱，第一反应就是“是不是一个容器”。这当然是一部分，但在理解层面上，可以先把它看得更简单：**系统会先规定，Agent 到底能碰哪些文件和目录。**

最常见的一种思路就是：工作区里的内容可以碰，工作区外的系统位置不能随便碰；某些挂载进来的目录有明确边界；越界路径会被直接拦住。这就是文件边界。它解决的是：就算命令真的执行了，也不能让它随便在整台机器上乱逛。

再看网络边界。网络边界说的是：**命令就算成功运行了，它到底能不能访问外部网络，能访问哪些地方。** 这很重要，因为很多风险不在“命令跑没跑起来”，而在“它有没有把东西带出去”。所以一个完整的安全设计，通常不会只看本地文件，它还会看网络是否默认关闭、是否只允许访问特定目标、某些任务是否根本不该联网。

如果把这三层都放在同一个场景里看，会更好理解：

```text
模型想执行一个命令
→ 先看它有没有这个工具
→ 真执行时，只能在允许的目录里活动
→ 就算命令成功，也不一定能随便联网
```

所以“能执行”从来不等于“能随便执行”。

同一条命令，可能在工具策略层就被拦住；也可能在执行时被目录边界拦住；还可能在最后一步被网络边界拦住。它们挡的是不同位置的风险。

可以把这一层理解成一个三段式：

| 边界 | 它主要防什么 |
|------|--------------|
| 工具可见性 | 不该给模型看的工具，先别给 |
| 文件边界 | 就算执行了，也别乱碰系统目录 |
| 网络边界 | 就算执行了，也别随便往外连 |

这三层不是重复，而是互相补位。因为“看得见”“跑得动”“能联网”本来就是三件不同的事。

### 2.4 `sandbox`、`tool policy` 和 `elevated`，其实是三回事

这一块特别容易被混成一团，所以先用一句话分清：

| 概念 | 它主要在管什么 |
|------|----------------|
| `tool policy` | 模型能不能看见和调用某个工具 |
| `sandbox` | 命令具体在什么边界里运行 |
| `elevated` | 极少数情况下，能不能临时申请更高权限 |

先说 `sandbox`。它更像运行现场本身，也就是这条命令到底是在 host 上跑，还是在受控环境里跑；能看到哪些目录；网络是什么模式。

这里顺手解释一下 `host`。它可以先理解成“你当前这台真实机器的环境”。所以“在 host 上跑”和“在沙箱里跑”的差别，不在于命令长什么样，而在于它最终落在哪个边界里执行。

再说 `tool policy`。它更像发门票。门票没发，连入场都谈不上。

最后是 `elevated`。这个词最容易让人误会成“万能钥匙”，其实不是。更准确地说，它像一张**临时通行证**。只有在某些本来不该默认放开的场景下，系统才会考虑让它出现。

所以：

- `elevated` 不是日常路径；
- 它不是为了偷懒而存在；
- 它更不是“安全配置搞不定时一键全开”的兜底按钮。

它存在的意义，是处理那些“默认边界是对的，但眼下这一步确实需要更高权限，而且需要被显式控制”的情况。这时通常会伴随几个特征：需要明确开关，需要受来源约束，常常和会话级状态或审批挂钩，用完就应该回到正常边界。

可以把这三者的关系再压成一句更顺口的话：

- `tool policy` 决定“看不看得见”；
- `sandbox` 决定“在哪儿跑、能碰哪儿”；
- `elevated` 决定“要不要临时升一级”。

把这三个层次分清楚以后，很多配置项就不会再看成一团了。

也就是说，`elevated` 的重点不是“让系统更自由”，而是“让升级这件事本身被控制住”。

如果把第二节收成一句话，就是：**OpenClaw 的安全不是靠单一开关实现的，而是先收窄工具可见性，再控制执行条件，再加上文件、网络和升级通道的边界，一层层把能力圈起来。**

## 第三节 知道机制以后我们应该如何做

前两节讲的是“为什么会有这些门”和“这些门怎么工作”。但真正开始配置时，很多同学最关心的其实是：**那我到底该怎么起步，才不至于一上来就把系统弄得太死，或者太危险？**

这一节就回答这个问题。

### 3.1 第一步不是追求最强，而是先得到一个可观察的默认配置

对初学者来说，最怕的不是系统不够强，最怕的是你根本不知道它下一步会做什么。所以刚开始配置时，目标不要写成“让它尽可能自动”，更稳的目标应该是：**让它的行为先变得可观察、可预测、可追溯。**

通常一个更稳的起步思路会像这样：

| 项目 | 起步建议 |
|------|----------|
| 工具范围 | 先只给当前任务真正需要的几类工具 |
| `tools.profile` | 先用保守套餐，不要一上来就全给 |
| `tools.deny` | 把明显高风险、当前又用不到的先禁掉 |
| `security` | 从 `allowlist` 起步 |
| `ask` | 初期用 `always` 或 `on-miss` |

这套思路背后其实只有一句话：先把边界变窄，等你观察够了，再一点点放宽。很多人一开始就想跳到 `full + off`，这在极少数高度受控环境里也许没问题，但对大多数刚开始使用的人来说，这通常不是“省事”，而是“把本来该逐步建立的信任一次性透支掉”。

如果你愿意把这个过程想成三个阶段，也可以这样记：

| 阶段 | 目标 | 更像什么 |
|------|------|----------|
| 观察期 | 看清 Agent 平时会怎么做 | 少放权，多确认 |
| 建立信任期 | 把常见安全命令沉淀下来 | 逐步扩充 allowlist |
| 稳定协作期 | 让日常任务更顺滑 | 只在新情况上保留确认 |

这样做的好处是，你不会急着一次性“毕业”。你会允许自己的边界随着使用经验一起成熟。

### 3.2 用好 `allow`、`deny` 和工具分组，先把工具面收窄

如果你只记一个最实用的安全原则，我建议记这个：**不是任务需要什么你就临时拦什么，而是先想清楚任务根本不需要什么。**

这个思路会让配置轻松很多。原因很简单，“允许什么”通常比“禁止一切可能的坏事”更容易想清楚。所以更好的做法往往是：先从一个较小的 profile 开始；用 `allow` 补少量必须的能力；再用 `deny` 把高风险或当前无关的部分明确挡掉。

这时“工具组”就很有用，因为它能让你按类别收口，而不是逐个盯名字。例如，从思路上你会更容易判断：当前任务只需要文件和只读查看类工具；当前任务不需要网络相关能力；当前任务不需要更高权限的执行能力。这比面对一长串具体工具名更容易上手。

这里有一个经验非常实用：如果你一时拿不准某个工具该不该给，不妨先问自己一句话：

**“没有它，这个任务会完全做不下去吗？”**

如果答案是否定的，那通常没必要在第一轮就把它放进来。

安全配置最怕的不是少给一个暂时用不到的工具，而是多给了一个本来不该出现的高风险工具。

这里还有一个非常实用的提醒：**别把 `TOOLS.md` 和真正的工具权限混在一起。** `TOOLS.md` 更像环境备注，真正决定“模型到底看见什么”的，还是这章讲的工具策略和运行时边界。

### 3.3 用审批和 allowlist 建立“可积累的信任”

很多人不喜欢审批，觉得每次确认很烦。这个感觉完全可以理解。但如果换个角度看，审批不是“系统不够聪明”的表现，它更像一个学习阶段。

尤其是刚开始和 Agent 协作时，你其实并不知道：它做这类任务通常会走哪些命令；哪些命令你觉得没问题；哪些边界你希望它永远别碰。所以这时的审批，不只是拦，它也在帮你认识系统。

而 `allowlist` 的价值就在这里。它把“这次确认过了”慢慢积累成“以后同类情况默认可走”。所以比较稳的成长路径通常是：

```text
先观察
→ 再积累 allowlist
→ 再减少不必要的询问
→ 最后形成适合自己的默认边界
```

这条路比一上来全放开更慢一点，但它建立起来的是长期可复用的信任，而不是一次性的侥幸。

如果你问“什么时候适合 `on-miss` 这种模式”，答案通常是：**当你已经对常见命令有基本认知，但仍希望对新情况保留一道确认时。** 这也是为什么它经常适合作为日常协作的中间态。它没有 `always` 那么吵，也没有 `off` 那么冒险。

这里还可以再补一个判断角度：不是所有任务都值得同样程度的信任。

如果一个任务基本只是读取、搜索、查看日志，那你通常可以更放松一些。

如果一个任务会改文件、装依赖、发网络请求，或者碰到删除、覆盖、迁移这类动作，就应该自然地提高警惕。

也就是说，信任不是只看“这个 Agent 我熟不熟”，还要看“它眼下这一步到底要做什么”。

### 3.4 建一份自己的安全使用清单，排障时按顺序看门

最后给一个非常实际的建议：不要把安全使用只理解成“配完一次就结束”。更好的做法是，给自己留一份很短的检查清单。

比如：

| 检查项 | 你要问自己的问题 |
|--------|------------------|
| 工具面 | 这次任务真的需要这么多工具吗？ |
| 写权限 | 它会不会改到我不想动的地方？ |
| 网络 | 这个任务需要联网吗？需要连到哪里？ |
| 审批 | 这一步是不是不可逆，是否值得保留人工确认？ |
| 升级 | 我现在真的需要 `elevated` 吗？ |

这份清单的意义不是让你变得神经紧张，而是让你每次放权时都更有意识。

如果你还想再把判断变得更简单，可以顺手记一个“可逆性”视角。所谓可逆性，就是这一步做错了以后，收回来到底难不难。

| 动作类型 | 例子 | 一般可以怎么对待 |
|----------|------|------------------|
| 只读动作 | 搜索、查看、统计、读日志 | 相对放松 |
| 可恢复动作 | 新建文件、在受控目录写临时结果 | 可以谨慎放行 |
| 难恢复动作 | 删除、覆盖关键配置、向外发送数据 | 尽量保留确认 |

这个视角很好用，因为它不要求你死记每个命令名，而是先看“这一步错了以后能不能收回来”。

另外，排障时也建议按“门”的顺序查。很多同学遇到 `exec` 跑不起来，第一反应就是：“是不是命令写错了？” 当然有可能，但很多时候，真正的原因其实在更前面。

一个很实用的排障顺序是：

1. 这个工具是不是根本没进当前可见工具集？
2. 它是不是被 `deny` 或 profile 策略收掉了？
3. 它是不是卡在 `security` 规则上？
4. 它是不是在等审批？
5. 它是不是被文件边界或网络边界拦住了？
6. 它是不是试图走一个不该默认走的 `elevated` 通道？

这套顺序的好处是，你会先查系统边界，再查命令本身。这样通常更快，也更不容易误判。

这里还有一个很重要的排障习惯：**不要为了“先跑起来看看”就直接把边界全开。**

很多人一着急，就想把 `deny` 改成 `full`，把审批关掉，再试一次。

这样当然可能让问题暂时消失，但你也同时失去了判断“到底是哪一层在拦”的机会。

更稳的做法是一次只放开一层。这样你不仅能把问题修掉，还能真正知道系统原来卡在什么地方。

如果把第三节收成一句话，就是：**最好的安全使用方式，不是靠胆大，而是靠先收窄、再观察、再逐步放开，让能力和信任一起长出来。**

---

## 本章小结

这一章真正讲清楚了三件事。

第一，为什么 OpenClaw 必须有安全沙箱。因为只要 Agent 真的能动手，风险就不再只是“回答错了”，而是可能真的碰坏文件、进程和网络边界。

第二，安全不是一个单独开关，而是一条链：

| 机制 | 它主要负责什么 |
|------|----------------|
| `tools.profile` / `allow` / `deny` | 先决定模型能看见哪些工具 |
| `security` / `ask` | 决定命令能不能执行、要不要先确认 |
| `allowlist` | 把已知可信的命令沉淀成长期规则 |
| `safeBins` | 给少量相对只读的小工具提供更轻的通道 |
| 文件与网络边界 | 限制命令能碰哪里、能连哪里 |
| `elevated` | 用受控方式处理少数必须升级的场景 |

第三，真正好用的安全，不是一下子把边界全拆掉，而是先从保守配置起步，再根据真实使用一点点放开。这套思路听上去没有“全自动”那么刺激，但它才是长期协作真正站得住的路。因为你不是在赌系统这次会不会聪明，你是在给系统搭一套，即使出错也收得住的边界。

这就是“护栏不是枷锁”的真正意思。护栏的价值，不是让车开不动，而是让你敢把车开上真正的路。

真正成熟的协作，从来不是“毫无保留地全信”，而是“边界清楚地放心交给它做”。
